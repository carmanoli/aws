Lab 31 - AWS CloudFormation Change Sets

Objective

The objective of this lab is to learn how AWS CloudFormation Change Sets allow administrators to preview infrastructure changes before applying them to a running stack.

Instead of updating a stack immediately, CloudFormation generates an execution plan showing exactly which resources will be modified, replaced, created or deleted.

Part 1 - Understanding Change Sets

A Change Set is a preview of the actions CloudFormation will perform before updating an existing stack.

Rather than modifying resources immediately, CloudFormation first compares the existing stack with the updated template and produces an execution plan.

In production environments, infrastructure teams commonly review Change Sets before approving deployments.

Part 2 - First Change Set

A new Output was added to the template: 

StackLastUpdated:
  Description: Demonstration output for Change Sets
  Value: "Lab31"

A Change Set was created using the updated template.

CloudFormation reported: 

Resource Changes: 0

No changes
There are no resources to be changed

Explanation

Outputs are not CloudFormation resources. Changing Outputs does not modify the deployed infrastructure. Therefore no resource changes are reported.

Part 3 - Creating a Real Resource Change

To demonstrate an actual infrastructure modification, the following property was changed: 

HealthCheckGracePeriod:
120

Changed to

HealthCheckGracePeriod:
150

A new Change Set was created.

CloudFormation reported:
Property Value
Logical ID LabWebAutoScalingGroup
Action Modify
Resource Type AWS::AutoScaling::AutoScalingGroup
Replacement False

Understanding the Result

The Change Set confirmed that:

  • The Auto Scaling Group would be modified.
  • No new resources would be created.
  • No resources would be deleted.
  • No replacement would occur.
  • The EC2 instance would remain unchanged.
  • The Load Balancer would remain attached.

This preview allows administrators to evaluate deployment impact before execution.

Executing the Change Set

After reviewing the proposed modifications, the Change Set was executed.

CloudFormation updated the existing Auto Scaling Group in place without recreating any infrastructure resources.

Verifying the Update

The updated property can be verified using:

  • CloudFormation Template
  • EC2 Console → Auto Scaling Groups → Details
  • AWS CLI

CLI example:

aws autoscaling describe-auto-scaling-groups \
--auto-scaling-group-names lab-cfn-web-asg
Expected output: 
"HealthCheckGracePeriod": 150

Update Stack vs Change Set

Update Stack Change Set
Applies changes immediately. Creates a deployment preview.
No opportunity to inspect changes. Allows review before execution.
Higher operational risk. Safer deployment process.
Suitable for development. Recommended for production.

Enterprise Best Practices

  • Always create a Change Set before updating production infrastructure.
  • Review every resource marked for modification.
  • Pay particular attention to the Replacement column.
  • Avoid executing a Change Set that replaces critical infrastructure without planning.
  • Use Change Sets together with version-controlled CloudFormation templates.

Conclusion

In this lab, Change Sets were introduced as a safe deployment mechanism for AWS CloudFormation. Instead of updating infrastructure immediately, CloudFormation generated a detailed execution plan showing the exact impact of the proposed changes.

The lab demonstrated both a Change Set with no infrastructure modifications (Outputs only) and a Change Set affecting an Auto Scaling Group without requiring resource replacement. This workflow reflects industry best practices for controlled infrastructure deployments using Infrastructure as Code.
