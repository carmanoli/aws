Lab 36 - CloudFormation UpdateReplacePolicy

AWS learning path - Infrastructure as Code, resource replacement and lifecycle protection

1. Lab Overview

This lab focused on the UpdateReplacePolicy attribute in AWS CloudFormation. The goal was to understand what happens to an old physical resource when a stack update requires CloudFormation to replace that resource.

The lab was designed as a temporary, low-cost exercise. It did not modify the permanent production-like lab infrastructure managed by the existing Auto Scaling Group stack.

2. Objectives

  • Understand the purpose of UpdateReplacePolicy.
  • Identify when CloudFormation performs a resource replacement.
  • Use Change Sets to validate whether an update will cause Replacement: True.
  • Demonstrate UpdateReplacePolicy: Delete.
  • Demonstrate UpdateReplacePolicy: Retain.
  • Understand when UpdateReplacePolicy: Snapshot can be used.
  • Compare UpdateReplacePolicy with DeletionPolicy.
  • Clean up all temporary resources created during the lab.

4. Concept: What is UpdateReplacePolicy?

UpdateReplacePolicy is a CloudFormation resource attribute that controls what happens to the old physical resource when an update operation requires replacement.

During some stack updates, CloudFormation cannot modify a resource in place. Instead, it creates a new physical resource, updates the logical resource mapping, and then processes the old physical resource according to the configured UpdateReplacePolicy.

Possible behaviours

Policy Behaviour Typical use case
Delete The old resource is deleted after replacement. Temporary or stateless resources.
Retain The old resource is kept but removed from CloudFormation management. Resources that must not be accidentally destroyed.
Snapshot A snapshot is created before the old resource is deleted. Storage resources such as EBS volumes or databases, when supported.

5. UpdateReplacePolicy vs DeletionPolicy

These two attributes are related but they act in different situations.

Attribute When it applies Example scenario
DeletionPolicy When a resource is removed from the template or when the stack is deleted. Deleting a stack that contains an EBS volume.
UpdateReplacePolicy When a stack update causes a resource replacement. Changing a property that requires a new EC2 instance to be created.
The most important rule from this lab: UpdateReplacePolicy only matters when the Change Set shows Replacement: True.

6. Temporary EC2 Template

To demonstrate replacement clearly, a temporary EC2 instance was created using CloudFormation. The stack was named: 

lab36-updatereplacepolicy-ec2

The initial template used UpdateReplacePolicy: Delete and DeletionPolicy: Delete. 

AWSTemplateFormatVersion: '2010-09-09'
Description: Lab 36 - UpdateReplacePolicy using EC2 replacement

Resources:

  Lab36SecurityGroup:

    Properties:
      GroupDescription: Lab36 temporary Security Group
      VpcId: vpc-0a2830bdd2b63ed4e

  Lab36Instance:
    Type: AWS::EC2::Instance
    UpdateReplacePolicy: Delete
    DeletionPolicy: Delete
    Properties:
      ImageId: ami-0402e980e69d5978b

      SubnetId: subnet-0e0cd186d22846c61
      SecurityGroupIds:
        - !Ref Lab36SecurityGroup
      Tags:
        - Key: Name
          Value: lab36-updatereplacepolicy-delete

7. Test 1 - UpdateReplacePolicy Delete

The first test used: 

UpdateReplacePolicy: Delete
DeletionPolicy: Delete

The subnet was changed to force a replacement of the EC2 instance: 

SubnetId: subnet-0d6b5a638e1b4b6f8

The Change Set showed: 

Lab36Instance
Action: Modify
Resource type: AWS::EC2::Instance
Replacement: True
Policy action: ReplaceAndDelete

Observed CloudFormation events

Lab36Instance UPDATE_COMPLETE
lab36-updatereplacepolicy-ec2 UPDATE_COMPLETE_CLEANUP_IN_PROGRESS
Lab36Instance DELETE_IN_PROGRESS
Lab36Instance DELETE_COMPLETE
lab36-updatereplacepolicy-ec2 UPDATE_COMPLETE
Result: the old EC2 instance was deleted after the replacement. This validated UpdateReplacePolicy: Delete.

8. Test 2 - UpdateReplacePolicy Retain

The second test changed the policy to: 

UpdateReplacePolicy: Retain
DeletionPolicy: Delete

The template was updated as follows: 

AWSTemplateFormatVersion: '2010-09-09'
Description: Lab 36 - UpdateReplacePolicy using EC2 replacement

Resources:

  Lab36SecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Lab36 temporary Security Group
      VpcId: vpc-0a2830bdd2b63ed4e

  Lab36Instance:
    Type: AWS::EC2::Instance
    UpdateReplacePolicy: Retain
    DeletionPolicy: Delete
    Properties:
      ImageId: ami-0402e980e69d5978b
      InstanceType: t3.micro
      SubnetId: subnet-0e0cd186d22846c61
      SecurityGroupIds:
        - !Ref Lab36SecurityGroup
      Tags:
        - Key: Name
          Value: lab36-updatereplacepolicy-retain

The subnet change again caused a replacement. During cleanup, CloudFormation reported: 

lab36-updatereplacepolicy-ec2 UPDATE_COMPLETE_CLEANUP_IN_PROGRESS
Lab36Instance DELETE_SKIPPED
lab36-updatereplacepolicy-ec2 UPDATE_COMPLETE
Result: the old EC2 instance was retained. CloudFormation skipped deletion because UpdateReplacePolicy: Retain was configured.

9. Understanding DELETE_SKIPPED

The event DELETE_SKIPPED means that CloudFormation did not delete the old physical resource during replacement.

However, the retained resource is no longer managed by the stack. It still exists in the AWS account, but it is now the responsibility of the operator to track and delete it when it is no longer needed.

In real environments, retained resources can become orphaned resources. This can create unnecessary cost and operational risk if they are not tracked.

10. Identifying Resources Managed by CloudFormation

CloudFormation adds reserved tags to many supported resources. These tags help identify whether a resource was created by a stack: 

aws:cloudformation:stack-name
aws:cloudformation:stack-id
aws:cloudformation:logical-id

The most reliable way to identify the currently managed physical resource is:

  1. Open the CloudFormation stack.
  2. Go to the Resources tab.
  3. Find the logical resource Lab36Instance.
  4. Compare the physical ID with the EC2 instance IDs.

A retained instance may still show CloudFormation-related tags, but it is no longer the current physical resource controlled by the stack.

11. Snapshot Behaviour

The Snapshot option is only supported for resource types that support snapshots. Examples include:

  • AWS::EC2::Volume
  • AWS::RDS::DBInstance
  • AWS::RDS::DBCluster
  • Other storage-related resources that explicitly support snapshots

In this lab, a small EBS volume was temporarily created to investigate the snapshot scenario. However, the attempted update did not produce Replacement: True in the Change Set. 

Lab36Volume
Action: Modify
Resource type: AWS::EC2::Volume
Replacement: False

Because the Change Set showed Replacement: False, the UpdateReplacePolicy: Snapshot attribute would not be triggered. The Change Set was therefore deleted and the temporary stack was removed.

This was an important practical lesson: do not assume that a property change will cause replacement. Always verify the Change Set before executing the update.

The temporary snapshot stack was named:

14. Key Lessons Learned

  • UpdateReplacePolicy controls the old physical resource only during a replacement update.
  • If the Change Set shows Replacement: False, UpdateReplacePolicy is not relevant for that update.
  • Delete removes the old resource after replacement.
  • Retain preserves the old resource but removes it from stack management.
  • Snapshot is only available for supported storage resources.
  • Retained resources must be tracked manually to avoid orphaned infrastructure and unnecessary costs.
  • Change Sets are essential before executing CloudFormation updates in professional environments.

15. Enterprise Best Practices

  • Always review Change Sets before executing updates in production.
  • Use DeletionPolicy and UpdateReplacePolicy together for important stateful resources.
  • Use Retain carefully, because it transfers responsibility from CloudFormation to the operations team.
  • Use Snapshot for supported storage resources where data recovery is required.
  • Document retained resources and include cleanup procedures.
  • Avoid testing lifecycle policies directly on production stacks.
  • Use temporary stacks to validate behaviour safely.

16. Final Summary

In this lab, CloudFormation UpdateReplacePolicy was studied and validated using temporary resources. The lab successfully demonstrated Delete and Retain behaviours during EC2 instance replacement.

The lab also showed that Snapshot requires both a supported resource type and an actual replacement event. Since the EBS volume update did not produce Replacement: True, the snapshot behaviour was not executed.

The most valuable practical outcome was the correct use of Change Sets as the source of truth before running CloudFormation updates. This is a professional practice that reduces the risk of unexpected resource replacement, data loss, and orphaned infrastructure.
