Avoid deletion of Critical Resources:

How to protect databases (RDS) or storage buckets (S3) from being accidentally destroyed when a Stack is deleted? The answer involves setting the DeletionPolicy attribute: Retain

CloudFormation Drift Detection:

How to detect when the actual state of resources in your AWS environment deviates from the expected state defined in your CloudFormation template? The answer involves using the Drift Detection feature to compare the current state with the expected state and identify any discrepancies.

Boas Práticas AWS CDK / IaC:

Acoplamento entre StacksO Problema (Anti-padrão):Criar dependências diretas entre Stacks diferentes (usando referências cruzadas ou Export / Fn::ImportValue). Se alterares ou tentares eliminar uma Stack que fornece um dado (output) para outra, a AWS bloqueia a atualização. Isto gera um bloqueio circular difícil de resolver (exige múltiplos deploys de "limpeza").A Solução (Boa Prática):Manter as Stacks 100% independentes e desacopladas. Para passar variáveis ou dados de configuração de uma Stack para outra, deve usar-se um intermediário.Como implementar:Stack A (Origem): Guarda o valor gerado (ex: o ID de uma rede ou base de dados) no AWS Systems Manager (SSM) Parameter Store.Stack B (Destino): Lê o valor diretamente do caminho do Parameter Store no momento do deploy.Resultado:As Stacks tornam-se autónomas. Podes atualizar, destruir ou recriar a Stack de origem sem o risco de corromper ou bloquear a Stack de destino.

Nested Stacks

Se a tua infraestrutura faz toda parte do mesmo projeto e queres fazer o deploy de tudo com um único comando, as Nested Stacks são a melhor alternativa para organizar o código. Se os projetos forem independentes, continua a usar o Parameter Store.
