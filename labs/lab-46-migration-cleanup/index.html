AWS Lab 46 – Migration Cleanup and Infrastructure Consolidation

Objective

Safely validate the migration from the legacy Default VPC infrastructure to the new Custom VPC infrastructure and remove obsolete resources while maintaining service availability.

Initial Environment

  • Legacy CloudFormation stack: lab26-asg-stack
  • Production CloudFormation stack: lab38-custom-vpc-stack
  • Application already migrated to a Custom VPC.
  • Instances running in private subnets behind an Internet-facing Application Load Balancer.

Validation Performed

  1. Verified the Application Load Balancer DNS of the new environment.
  2. Confirmed the Target Group reported healthy targets.
  3. Verified the Auto Scaling Group deployed instances only in private subnets.
  4. Confirmed the application was served by the new infrastructure.

Legacy Infrastructure Audit

The remaining legacy resources were identified:

  • Application Load Balancer: lab-web-alb
  • Target Group: lab-web-tg
  • Launch Template: launch-web-template
  • Legacy Security Groups in the Default VPC.

CloudFormation Cleanup

  • Validated that the new environment was fully operational.
  • Deleted the legacy CloudFormation stack.
  • Confirmed that the stack deletion left orphaned resources.
  • Removed the obsolete ALB and Target Group manually.

Enterprise Best Practices

  • Validate production before decommissioning legacy resources.
  • Use Infrastructure as Code as the authoritative configuration.
  • Perform post-migration audits to detect orphan resources.
  • Remove unused resources to reduce operational cost.
  • Maintain high availability during migrations.
