Lab 25 - CloudFormation with Launch Template

1. Lab Objective

The objective of this lab was to extend the previous CloudFormation fundamentals lab by creating an Amazon EC2 instance through AWS CloudFormation while reusing an existing EC2 Launch Template.

Instead of defining the full EC2 configuration directly inside the CloudFormation template, the stack reused the existing Launch Template named launch-web-template. This approach avoids duplicating configuration such as AMI selection, User Data, IAM role configuration, Apache installation, Git deployment logic and other instance bootstrap settings.

2. Services Used

  • AWS CloudFormation
  • Amazon EC2
  • EC2 Launch Templates
  • Application Load Balancer
  • Target Group
  • Security Groups
  • AWS Systems Manager Session Manager
  • Apache HTTP Server
  • Git / GitHub deployment through User Data

3. Existing Resources Reused

Resource Name Purpose
Launch Template launch-web-template Provides the EC2 base configuration and website deployment automation.
Application Load Balancer lab-web-alb Receives HTTP traffic from the internet.
Target Group lab-web-tg Routes traffic from the ALB to healthy EC2 instances.
ALB Security Group lab-alb-sg Allows inbound HTTP traffic from the internet.
EC2 Security Group lab-ec2-web-sg Allows HTTP traffic only from the ALB Security Group.
IAM Role lab-ec2-ssm-role Allows EC2 administration through AWS Systems Manager Session Manager.

The CloudFormation-created EC2 instance was manually registered in the existing Target Group to verify that it could serve traffic through the existing Application Load Balancer.

5. Important Concept

CloudFormation only creates and configures the resources explicitly declared in the template. In this lab, the template created an EC2 instance and a Security Group, but it did not automatically register the instance in the existing Target Group because that association was not defined in the template.

This distinction is important in Infrastructure as Code. AWS resources are not implicitly connected just because they exist in the same environment. Relationships between services must be declared explicitly.

6. CloudFormation Template

The previous Lab 24 template was adapted so that the EC2 instance used the existing Launch Template. 

AWSTemplateFormatVersion: '2010-09-09'

Description: >
  Lab 25 - CloudFormation with Launch Template.
  Creates an EC2 instance using an existing Launch Template.

Parameters:

  VpcId:
    Type: AWS::EC2::VPC::Id
    Description: Existing VPC ID

  SubnetId:
    Type: AWS::EC2::Subnet::Id
    Description: Existing subnet for the EC2 instance

  LatestAmiId:
    Type: AWS::SSM::Parameter::Value<AWS::EC2::Image::Id>
    Default: /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64
    Description: Latest Amazon Linux 2023 AMI ID from AWS Systems Manager Parameter Store

Resources:

  LabSecurityGroup:
    Type: AWS::EC2::SecurityGroup
    Properties:
      GroupDescription: Security Group created by CloudFormation
      GroupName: lab-cfn-web-sg
      VpcId: !Ref VpcId
      SecurityGroupIngress:
        - IpProtocol: tcp
          FromPort: 80
          ToPort: 80
          CidrIp: 0.0.0.0/0

  LabEC2Instance:
    Type: AWS::EC2::Instance
    Properties:
      LaunchTemplate:
        LaunchTemplateName: launch-web-template
        Version: "2"
      SubnetId: !Ref SubnetId
      Tags:
        - Key: Name
          Value: lab-cfn-launch-template-ec2
        - Key: Lab
          Value: Lab25
        - Key: ManagedBy
          Value: CloudFormation

Outputs:

  SecurityGroupId:
    Description: Security Group created by the stack
    Value: !Ref LabSecurityGroup

  InstanceId:
    Description: EC2 Instance ID
    Value: !Ref LabEC2Instance

Note: The template still includes the LatestAmiId parameter from the previous lab, although the EC2 instance is now using the AMI defined in the Launch Template. In a later cleanup version, this parameter can be removed if it is no longer needed.

7. Issue Encountered

The first version of the updated template used the following syntax: 

LaunchTemplate:
  LaunchTemplateName: launch-web-template
  Version: $Latest

CloudFormation returned the following error: 

CloudFormation does not support using $Latest or $Default for LaunchTemplate version.
Please specify a version number or name.

8. Resolution

The Launch Template version was checked in the EC2 console under: 

EC2 > Launch Templates > launch-web-template > Versions

The active version was identified as version 2. The CloudFormation template was then updated to use an explicit version number: 

LaunchTemplate:
  LaunchTemplateName: launch-web-template
  Version: "2"

After deleting the failed stack in ROLLBACK_COMPLETE state, the stack was created again successfully.

9. Stack Creation

The stack was created with the following name: 

lab25-cfn-launch-template

Parameters selected during stack creation:

  • VpcId: default VPC
  • SubnetId: public subnet in the same VPC

The final stack status was:

CREATE_COMPLETE

10. Target Group Registration

After the stack was created, the new EC2 instance was manually registered in the existing Target Group: 

EC2 > Target Groups > lab-web-tg > Targets > Register targets

The new CloudFormation-created EC2 instance was selected and registered on port 80. After a short wait, the target became Healthy.

11. Access Model

The EC2 instance was not accessed directly through its public IP address. This was expected because the security model is designed so that the EC2 instances only accept HTTP traffic from the Application Load Balancer. 

Internet
   |
   v
ALB Security Group
   |
   v
Application Load Balancer
   |
   v
Target Group
   |
   v
EC2 Security Group
   |
   v
EC2 Instance

This is a recommended architecture because the EC2 instances are not directly exposed to the internet. Public traffic enters through the ALB, and the ALB forwards requests only to healthy registered targets.

12. Validation Through Session Manager

The new EC2 instance was accessed using AWS Systems Manager Session Manager, because SSH remains disabled in this environment.

The following commands were executed:

hostname

Example output:

ip-172-31-19-121.eu-north-1.compute.internal

The web server was then tested locally:

curl localhost

The command returned the HTML content of the website, confirming that Apache was running and that the website deployment from the Launch Template User Data had completed successfully.

13. Load Balancing Validation

To confirm that the Application Load Balancer was distributing traffic between the original EC2 instance and the CloudFormation-created EC2 instance, the website content was temporarily modified only on the CloudFormation-created instance.

The website directory was checked:

ls /var/www/html

The main HTML file was edited:

sudo nano /var/www/html/index.html

A temporary visible marker was added:

<h1>CloudFormation Instance</h1>

After refreshing the ALB DNS name in the browser several times, the page alternated between the original website and the modified page. This confirmed that the ALB was serving traffic from both instances.

14. Technical Conclusions

  • CloudFormation can create EC2 instances using existing Launch Templates.
  • When using a Launch Template in CloudFormation, the template version must be explicit.
  • $Latest and $Default are not accepted in this CloudFormation EC2 Launch Template configuration.
  • CloudFormation only creates the relationships declared in the template.
  • Registering an EC2 instance in a Target Group must be done explicitly, either manually or through the template.
  • Accessing EC2 instances through the ALB instead of directly through public IP addresses is a better security practice.
  • Session Manager allows administration without SSH access.
  • The Launch Template successfully reused the existing Apache and GitHub deployment automation.

15. Enterprise Relevance

This lab demonstrates the transition from manual resource creation to Infrastructure as Code. In a real enterprise environment, creating standalone EC2 instances for web workloads is less common than creating Auto Scaling Groups linked to Target Groups and Load Balancers.

However, this lab was important because it showed how CloudFormation can reuse an existing Launch Template and how explicit resource relationships are required when building infrastructure as code.

16. Cost Considerations

The lab used a small EC2 instance compatible with the Free Tier where available. However, because an additional EC2 instance was created, it is important to stop or delete unnecessary test resources after the lab if they are no longer needed.

The Application Load Balancer may generate costs depending on account credits and Free Tier conditions. Existing resources should be reused carefully and cleaned up when not needed.

17. Recommended Cleanup

If this CloudFormation-created EC2 instance is no longer required, delete the CloudFormation stack: 

CloudFormation > lab25-cfn-launch-template > Delete

Before deletion, the instance can also be deregistered from the Target Group if required: 

EC2 > Target Groups > lab-web-tg > Targets > Deregister

If the stack is deleted, the EC2 instance and the Security Group created by the stack are removed automatically, unless another resource dependency prevents deletion.

In this lab, an EC2 instance was created with CloudFormation using an existing Launch Template. The stack was successfully created after correcting the Launch Template version from $Latest to an explicit version number. The instance was manually added to the existing Target Group and validated through the Application Load Balancer. Session Manager was used for administration, maintaining the no-SSH security model already adopted in previous labs.
