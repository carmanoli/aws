Lab 19 - Amazon EventBridge Rules: Event-Driven Architecture

1. Lab Objective

The objective of this lab is to learn how to use Amazon EventBridge Rules to react to real AWS service events instead of scheduled events.

In the previous lab, EventBridge Scheduler was used to invoke a Lambda function on a recurring schedule. In this lab, EventBridge reacts to a real AWS API activity captured by AWS CloudTrail.

The final architecture is:

EC2 StopInstances API call
        ↓
AWS CloudTrail Trail
        ↓
Amazon EventBridge Rule
        ↓
AWS Lambda function
        ↓
Amazon CloudWatch Logs

3. Why This Lab Matters

Event-driven architectures are widely used in professional AWS environments. Instead of polling for changes, applications can react automatically when something happens.

Common use cases include:

  • Detecting when EC2 instances are stopped or started.
  • Triggering automated remediation workflows.
  • Sending notifications when infrastructure changes.
  • Starting serverless workflows after AWS API activity.
  • Auditing and reacting to security-relevant events.

4. Important AWS 2026 Note

During this lab, an important behavior was identified. Creating an EventBridge Rule from the AWS Console may create the rule in the normal ENABLED state. However, to process all CloudTrail management events, the rule may need to be enabled with the special state:

ENABLED_WITH_ALL_CLOUDTRAIL_MANAGEMENT_EVENTS

This setting was applied through the AWS CLI.

This was the key troubleshooting discovery in the lab. The rule pattern was correct, the Lambda target was correct, and CloudTrail was logging the event, but EventBridge did not match the event until the rule state was updated using the CLI.

5. Step 1 - Create the EventBridge Rule

Open the AWS Console and go to:

Amazon EventBridge → Rules → Create rule

Configure the rule details:

FieldValue
Namelab-lambda-change-rule
DescriptionTrigger Lambda when EC2 StopInstances API activity occurs
Event busdefault
StatusEnabled

Select the advanced builder if available.

6. Step 2 - Configure the Event Pattern

The final working event pattern used in this lab was based on the official AWS example for reacting to EC2 API calls through CloudTrail.

Use a custom JSON event pattern:

{
  "source": ["aws.ec2"],
  "detail-type": ["AWS API Call via CloudTrail"],
  "detail": {
    "eventSource": ["ec2.amazonaws.com"],
    "eventName": ["StopInstances"]
  }
}

This pattern means:

  • The event must come from Amazon EC2.
  • The event must be an AWS API call captured through CloudTrail.
  • The API source must be ec2.amazonaws.com.
  • The API operation must be StopInstances.

7. Step 3 - Select the Lambda Target

For the target, select:

FieldValue

Using Matched event is important because it sends the full EventBridge event to the Lambda function. This allows the Lambda function to inspect fields such as source, detail-type, eventName, and userIdentity.

8. Step 4 - Create a CloudTrail Trail

CloudTrail Event History alone was not enough for this lab. A CloudTrail Trail was created so that management events could be delivered and used by EventBridge.

Go to:

CloudTrail → Trails → Create trail

Configure the trail:

9. Step 5 - Update the Rule State with AWS CLI

The EventBridge rule had to be updated through AWS CLI to enable CloudTrail management events properly.

Open AWS CloudShell and run:

aws events put-rule \
  --name lab-lambda-change-rule \
  --event-bus-name default \
  --state ENABLED_WITH_ALL_CLOUDTRAIL_MANAGEMENT_EVENTS \
  --event-pattern '{
    "source": ["aws.ec2"],
    "detail-type": ["AWS API Call via CloudTrail"],
    "detail": {
      "eventSource": ["ec2.amazonaws.com"],
      "eventName": ["StopInstances"]
    }
  }'

The expected output is similar to:

{
    "RuleArn": "arn:aws:events:eu-north-1:ACCOUNT_ID:rule/lab-lambda-change-rule"
}

10. Step 6 - Generate the Real Event

To trigger the EventBridge rule, the EC2 instance must be stopped.

Go to:

EC2 → Instances → lab-web-server

If the instance is stopped, start it first. Wait until it reaches the Running state.

Then stop the instance:

Instance state → Stop instance

This generates the EC2 API call:

StopInstances

13. Troubleshooting Performed

This lab included real troubleshooting. The first attempt used Lambda configuration events, specifically:

UpdateFunctionConfiguration20150331v2

CloudTrail Event History showed the event, but EventBridge did not match it. The EventBridge Sandbox was then used to test event patterns with a valid EventBridge event structure.

The main lessons learned were:

  • CloudTrail Event History and EventBridge events are related but not presented in exactly the same format.
  • EventBridge Sandbox is useful to validate event patterns before relying on them in a rule.
  • The EventBridge Console may not expose every state or behavior available in the API.
  • AWS CLI is essential for troubleshooting and professional cloud operations.
  • CloudTrail management events required the rule state ENABLED_WITH_ALL_CLOUDTRAIL_MANAGEMENT_EVENTS.

14. Disable the Rule After the Lab

To avoid unnecessary invocations, the rule was disabled at the end of the lab.

The AWS Console did not allow disabling the rule directly, so AWS CLI was used:

aws events disable-rule \
  --name lab-lambda-change-rule \
  --region eu-north-1

To verify the state:

aws events describe-rule \
  --name lab-lambda-change-rule \
  --region eu-north-1

The expected result is:

"State": "DISABLED"

15. Cost Considerations

  • The EC2 instance was stopped after testing to avoid compute costs.
  • The EventBridge rule was disabled to avoid unnecessary Lambda invocations.
  • The CloudTrail Trail writes logs to S3, so small S3 storage costs may apply over time.
  • Only write management events were enabled to reduce event volume.
  • Data events and Insights events were disabled because they are not required for this lab and may increase costs.

If this lab environment is not needed for future labs, consider stopping CloudTrail logging or deleting the trail and its S3 bucket after reviewing the logs. If future labs will continue using CloudTrail and EventBridge, keeping the trail can be useful.

17. Key Takeaways

  • EventBridge Rules are used to react to real events from AWS services.
  • EventBridge Scheduler should be used for scheduled tasks.
  • CloudTrail is required for AWS API Call events.
  • Event patterns must be precise and should be tested.
  • CloudWatch Logs are essential to validate serverless event-driven workflows.
  • AWS CLI is often required when the Console does not expose advanced behavior.
