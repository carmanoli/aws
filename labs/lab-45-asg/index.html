Lab 45 - Auto Scaling Group Migration to the Custom VPC

1. Objective

The objective of this lab was to migrate the application layer from the Default VPC to the Custom VPC by replacing a temporary EC2 instance with an Auto Scaling Group deployed in private subnets.

2. Initial Architecture

Before this lab, the Custom VPC already contained public subnets, private subnets, route tables, an Application Load Balancer, a target group, VPC Flow Logs, and a temporary EC2 instance used to validate web traffic through the ALB.

3. Target Architecture

Internet
   |
Application Load Balancer
   |
Target Group
   |
Auto Scaling Group
   |
Private Subnet A / Private Subnet B

4. Technical Decisions

The EC2 instances were placed in private subnets because they do not need direct public access from the Internet. The Application Load Balancer remains in public subnets and forwards HTTP traffic to the instances through the target group.

This follows a common enterprise architecture pattern where only the load balancer is Internet-facing, while application servers remain isolated in private networks.

5. CloudFormation Resources Added

The following resources were added to the existing CloudFormation stack:

  • A new EC2 Launch Template
  • A new Auto Scaling Group

Launch Template

LabAutoScalingLaunchTemplate:
  Type: AWS::EC2::LaunchTemplate
  Properties:
    LaunchTemplateName: lab-web-launch-template
    LaunchTemplateData:
      ImageId: !Ref LatestAmiId
      InstanceType: t3.micro
      IamInstanceProfile:
        Name: !Ref LabEC2InstanceProfile
      SecurityGroupIds:
        - !Ref LabWebSecurityGroup
      TagSpecifications:
        - ResourceType: instance
          Tags:
            - Key: Name
              Value: lab-web-asg-instance
      UserData:
        Fn::Base64: !Sub |
          #!/bin/bash
          dnf update -y
          dnf install -y httpd
          systemctl enable httpd
          systemctl start httpd
          echo "<h1>Lab 45 - Auto Scaling Group</h1>" > /var/www/html/index.html
          echo "<p>Served by Auto Scaling in Private Subnets</p>" >> /var/www/html/index.html
          echo "<p>Hostname: $(hostname)</p>" >> /var/www/html/index.html

Auto Scaling Group

LabAutoScalingGroup:
  Type: AWS::AutoScaling::AutoScalingGroup
  Properties:
    AutoScalingGroupName: lab-custom-vpc-asg
    VPCZoneIdentifier:
      - !Ref LabPrivateSubnetA
      - !Ref LabPrivateSubnetB
    LaunchTemplate:
      LaunchTemplateId: !Ref LabAutoScalingLaunchTemplate
      Version: !GetAtt LabAutoScalingLaunchTemplate.LatestVersionNumber
    MinSize: 1
    MaxSize: 2
    DesiredCapacity: 1
    TargetGroupARNs:
      - !Ref LabAlbTargetGroup
    HealthCheckType: ELB
    HealthCheckGracePeriod: 120
    Tags:
      - Key: Name
        Value: lab-web-asg-instance
        PropagateAtLaunch: true

7. Troubleshooting

After deployment, the instance created by the Auto Scaling Group initially became unhealthy in the target group.

Observed Symptoms

  • The target group health check failed.
  • The Auto Scaling instance appeared as unhealthy.
  • The instance was not managed by Systems Manager.

Console Output Analysis

The EC2 console output showed that the instance failed during User Data execution. The main errors were:

Failed to download metadata for repo 'amazonlinux'
Status code: 403
No match for argument: httpd
Failed to start httpd.service: Unit httpd.service not found
SSM Agent unable to acquire credentials

Root Cause

The problem was not the Auto Scaling Group, the Launch Template, the ALB, or the security groups. The root cause was the S3 Gateway Endpoint policy created in a previous lab.

The policy was too restrictive and only allowed access to the lab S3 bucket. Amazon Linux repositories are also accessed through S3. Because the private subnet route table used the S3 Gateway Endpoint, repository access was routed through the endpoint and denied by the restrictive policy.

Resolution

The S3 Gateway Endpoint policy was temporarily relaxed to allow broader S3 access:

PolicyDocument:
  Version: "2012-10-17"
  Statement:
    - Sid: AllowAllS3AccessFromVpcEndpoint
      Effect: Allow
      Principal: "*"
      Action: "s3:*"
      Resource: "*"

After the policy was updated, the Auto Scaling Group replaced the unhealthy instance. The new instance successfully installed Apache, started the web service, and passed the ALB health check.

8. Removal of the Temporary EC2 Instance

After the Auto Scaling Group became healthy, the temporary EC2 instance was removed from the template.

The manual target registration was also removed from the target group:

Targets:
  - Id: !Ref LabCustomVpcEC2Instance
    Port: 80

The Auto Scaling Group now automatically registers and deregisters instances in the target group.

10. Enterprise Relevance

This architecture is commonly used in production environments. The ALB is public, while application instances remain private. Auto Scaling improves availability and resilience by replacing unhealthy instances automatically.

The troubleshooting scenario was also realistic. In enterprise environments, restrictive endpoint policies, route tables, private networking, and bootstrap scripts often interact in unexpected ways.

11. AWS Well-Architected Framework Alignment

  • Security: Application instances are placed in private subnets and only accept traffic from the ALB security group.
  • Reliability: Auto Scaling replaces unhealthy instances automatically.
  • Operational Excellence: CloudFormation is used as the single source of truth.
  • Cost Optimization: The architecture avoids unnecessary public EC2 exposure and keeps NAT Gateway disabled where not required.

14. Conclusion

Lab 45 completed the migration of the application layer into the Custom VPC. The environment now follows a production-style architecture where the ALB is public and the application instances are private, scalable, and managed by Auto Scaling.

The lab also demonstrated an important real-world troubleshooting case involving S3 Gateway Endpoint policies and Amazon Linux repository access.
