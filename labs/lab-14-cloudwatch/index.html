Understanding Namespaces

Each AWS service publishes its own metrics into CloudWatch.

Examples:

Service Namespace
EC2 AWS/EC2
EBS AWS/EBS
ALB AWS/ApplicationELB
S3 AWS/S3

CloudWatch acts as a centralized metrics repository.

Part 2 – Explore EC2 Metrics

Open

CloudWatch
→ Metrics
→ EC2
→ Per-Instance Metrics

Locate your running EC2 instance.

Select

CPUUtilization

A graph displaying CPU utilization should appear.

At idle, CPU usage is typically below:

1%

This is expected because only the operating system and Apache are running.

Understanding CPUUtilization

CPUUtilization represents the percentage of CPU currently being used.

Typical values:

CPU Meaning
0–10% Idle
20–50% Normal workload
70–80% High utilization
90–100% CPU saturation

Part 6 – Alarm States

A CloudWatch Alarm can be in three states.

INSUFFICIENT_DATA

CloudWatch does not yet have enough data.

Usually seen:

  • Immediately after alarm creation
  • When an instance starts
  • When metrics are unavailable

OK

The metric is within the configured threshold.

For this lab:

CPU ≤ 70%

ALARM

The threshold has been exceeded.

For this lab:

CPU > 70%

Part 7 – Generate CPU Load

Connect to the EC2 instance.

Update packages:

sudo dnf update -y

Install stress-ng:

sudo dnf install -y stress-ng

Verify installation:

stress-ng --version

Check the number of CPUs:

nproc

Example output:

2

Generate CPU load:

stress-ng --cpu 2 --timeout 600s

This command:

  • Uses both virtual CPUs
  • Runs for 10 minutes
  • Generates nearly 100% CPU utilization

Part 8 – Observe the Alarm

CloudWatch
→ Alarms

↓

↓

↓

This confirms the alarm is working correctly.

Part 9 – Explore Application Load Balancer Metrics

Open

CloudWatch
→ Metrics
→ ApplicationELB
→ Per AppELB Metrics

Useful metrics include:

  • RequestCount
  • TargetResponseTime
  • HTTPCode_Target_2XX_Count
  • HTTPCode_Target_3XX_Count
  • HTTPCode_Target_4XX_Count
  • HTTPCode_Target_5XX_Count
  • ActiveConnectionCount
  • NewConnectionCount
  • ProcessedBytes

Most Important ALB Metrics

RequestCount

Number of HTTP requests received.

Useful for:

  • Capacity planning
  • Traffic analysis

TargetResponseTime

Time required for the backend target to respond.

Useful for:

  • Detecting slow applications
  • Performance monitoring

HTTPCode_Target_2XX_Count

Successful responses.

Indicates normal application behavior.

HTTPCode_Target_4XX_Count

Client errors.

Examples:

  • 400
  • 401
  • 403
  • 404

Useful for identifying invalid requests or authentication problems.

HTTPCode_Target_5XX_Count

Server-side errors.

Examples:

  • 500
  • 502
  • 503
  • 504

One of the most important production monitoring metrics because it indicates application failures.

ActiveConnectionCount

Current open client connections.

Useful for understanding live traffic.

NewConnectionCount

Number of newly established client connections.

Useful for traffic trend analysis.

ProcessedBytes

Amount of data processed by the Load Balancer.

Useful for bandwidth monitoring.

CloudWatch Dashboards

CloudWatch Dashboards provide a single monitoring interface.

Example:

+-------------------------------------------------------+
| Requests/min      CPU EC2       Response Time          |
+-------------------------------------------------------+
| ALB 5XX Errors    Healthy Hosts  Network In            |
+-------------------------------------------------------+
| Auto Scaling      Disk Usage     Lambda Errors         |
+-------------------------------------------------------+

Dashboards allow operations teams to monitor an entire environment from one screen.

CloudWatch Integration

CloudWatch does not perform actions itself.

It evaluates metrics and changes alarm states.

Other AWS services react to these state changes.

Typical integrations include:

CloudWatch Alarm
        │
        ├── Amazon SNS
        ├── Auto Scaling
        ├── AWS Lambda
        ├── Systems Manager
        └── EventBridge

This makes CloudWatch the foundation of monitoring and automation within AWS.
