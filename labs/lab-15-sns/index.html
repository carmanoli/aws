# Lab 15 - Amazon SNS Integration with CloudWatch ## Objective In this lab, we integrate Amazon Simple Notification Service (SNS) with an existing Amazon CloudWatch Alarm to receive email notifications whenever the alarm changes state. By the end of this lab you will have: - Created an SNS Standard Topic - Created an email subscription - Confirmed the subscription - Integrated the existing CloudWatch Alarm with SNS - Received email notifications when the alarm enters the ALARM state - Received email notifications when the alarm returns to the OK state --- # Architecture ```text +----------------+ | EC2 | +----------------+ | | CPUUtilization | ▼ +------------------+ | CloudWatch Metric| +------------------+ | ▼ +------------------+ | CloudWatch Alarm | +------------------+ | ▼ +------------------+ | SNS Topic | +------------------+ | +-----------+-----------+ | | ▼ ▼ Email Notification Future Subscribers ``` --- # Existing Infrastructure The following resources were reused from previous labs: - EC2 Instance - CloudWatch Alarm - Application Load Balancer - Target Group - Launch Template - Security Groups - GitHub automatic deployment - Apache Web Server Existing CloudWatch Alarm: ``` lab-ec2-cpu-alarm ``` --- # Why Amazon SNS? Amazon SNS (Simple Notification Service) is a fully managed publish/subscribe messaging service. Instead of CloudWatch sending notifications directly to users, it publishes messages to an SNS Topic. Any number of subscribers can receive those messages. Typical subscribers include: - Email - SMS - AWS Lambda - Amazon SQS - HTTPS endpoints - AWS Chatbot - Mobile Push Notifications This decouples monitoring from notification delivery. --- # Step 1 - Create an SNS Topic Open: ``` AWS Console → SNS → Topics → Create topic ``` Configuration: | Setting | Value | |----------|-------| | Type | Standard | | Name | lab-cloudwatch-topic | Leave all other settings as default. Click: ``` Create topic ``` --- # Why Standard Instead of FIFO? CloudWatch alarm notifications do not require: - Message ordering - Deduplication - Exactly-once processing Therefore, a Standard Topic is the recommended option. FIFO topics are intended for transactional workloads and ordered message processing. --- # Step 2 - Create an Email Subscription Open: ``` SNS → Topics → lab-cloudwatch-topic → Create subscription ``` Configuration: | Setting | Value | |----------|-------| | Protocol | Email | | Endpoint | Your email address | Click: ``` Create subscription ``` The subscription status will initially be: ``` Pending confirmation ``` --- # Step 3 - Confirm the Subscription Open your email inbox. AWS sends a confirmation email similar to: ``` AWS Notification - Subscription Confirmation ``` Click: ``` Confirm subscription ``` Refresh the SNS console. The subscription status should now be: ``` Confirmed ``` --- # Step 4 - Integrate CloudWatch Alarm with SNS Open: ``` CloudWatch → Alarms → lab-ec2-cpu-alarm → Actions → Edit ``` Configure the notification for the ALARM state: | Setting | Value | |----------|-------| | Alarm State | In alarm | | Notification | Send to existing SNS Topic | | SNS Topic | lab-cloudwatch-topic | Save the changes. --- # Step 5 - Configure Notification for the OK State Edit the same alarm again. Configure: | Setting | Value | |----------|-------| | OK State | Send notification | | SNS Topic | lab-cloudwatch-topic | Save the changes. Now notifications will be sent whenever the alarm: - Enters ALARM - Returns to OK --- # Step 6 - Generate CPU Load Generate CPU load on the EC2 instance. Example: ```bash stress-ng --cpu 2 --timeout 600s ``` CloudWatch continuously monitors CPU utilization. Once the configured threshold is exceeded: ``` CloudWatch Alarm → ALARM ``` An email notification is sent. After CPU usage drops below the threshold: ``` CloudWatch Alarm → OK ``` A second email notification is sent. --- # Validation The following email notifications were successfully received: ``` ALARM: "lab-ec2-cpu-alarm" OK: "lab-ec2-cpu-alarm" ``` This validates the complete monitoring pipeline. --- # Final Architecture ```text EC2 Instance │ ▼ CloudWatch Metrics │ ▼ CloudWatch Alarm │ ▼ SNS Topic │ ▼ Email Notification ``` --- # Enterprise Best Practices Use Standard Topics for monitoring notifications. Use descriptive and reusable names. ``` lab-cloudwatch-topic ``` Separate monitoring from notification delivery. Reuse the same SNS Topic for multiple CloudWatch alarms whenever appropriate. Always configure notifications for both: - ALARM - OK This provides full visibility into incident lifecycle. Keep subscribers independent from publishers. CloudWatch should never know who receives notifications. SNS handles message distribution. --- # Benefits of SNS SNS enables one publisher to notify multiple consumers simultaneously. Future integrations can include: - AWS Lambda - Amazon SQS - EventBridge - AWS Chatbot - Slack - Microsoft Teams - SMS - HTTPS APIs No CloudWatch configuration changes are required when adding new subscribers. --- # AWS Services Used - Amazon EC2 - Amazon CloudWatch - Amazon SNS --- # Resources Created | Resource | Name | |----------|------| | SNS Topic | lab-cloudwatch-topic | | Email Subscription | Confirmed | --- # Resources Reused | Resource | Name | |----------|------| | EC2 Instance | lab-web-server | | CloudWatch Alarm | lab-ec2-cpu-alarm | | Launch Template | lab-web-template | | ALB | lab-web-alb | | Target Group | lab-web-tg | --- # Cost This lab is Free Tier friendly.
