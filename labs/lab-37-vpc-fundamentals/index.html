Lab 37 - Amazon VPC Fundamentals

AWS Networking Fundamentals using the existing lab infrastructure

1. Lab Overview

This lab introduces the fundamental concepts of Amazon Virtual Private Cloud (Amazon VPC). Instead of creating new resources, the objective is to inspect and understand the existing network where the current AWS infrastructure is already running.

The current environment includes an Application Load Balancer, an Auto Scaling Group, EC2 instances, Security Groups, CloudFormation, Systems Manager Session Manager, CloudWatch, SNS, Lambda and related services. All compute resources are currently deployed inside the default VPC in the Stockholm region.

2. Lab Objectives

  • Understand the role of an Amazon VPC.
  • Identify the VPC used by the existing Application Load Balancer.
  • Identify the subnets used by the current infrastructure.
  • Understand public subnets and private subnets.
  • Understand route tables and Internet Gateway routing.
  • Understand CIDR blocks using real values from the existing environment.
  • Compare the default VPC with a production-oriented custom VPC design.
  • Prepare for creating a custom VPC with CloudFormation in the next lab.

6. Availability Zones and Subnets

The ALB is associated with two subnets across two Availability Zones:

Availability Zone Subnet ID CIDR Block
eu-north-1a subnet-0e0cd186d22846c61 172.31.16.0/20

Using two Availability Zones improves fault tolerance. If one Availability Zone becomes unavailable, the load balancer can continue serving traffic through the other Availability Zone, assuming healthy targets are available there.

8. Route Table Analysis

The route table associated with both subnets is:

rtb-06e5e54ceb6256291

It contains the following routes:

Destination Target Status
172.31.0.0/16 local Active
0.0.0.0/0 igw-0a42945a4345f775c Active

8.1 Local Route

172.31.0.0/16 → local

This route allows resources inside the VPC to communicate with each other using private IP addresses. Traffic that belongs to the VPC CIDR block remains inside the VPC.

8.2 Internet Route

0.0.0.0/0 → igw-0a42945a4345f775c

This route sends all non-local traffic to the Internet Gateway. This is the key route that makes the associated subnets public.

Conclusion: The two subnets used by the ALB are public subnets because their route table contains a default route to an Internet Gateway.

9. Internet Gateway

The Internet Gateway identified in this lab is:

igw-0a42945a4345f775c

An Internet Gateway allows communication between resources in a VPC and the Internet, provided that route tables, public IP addressing, Security Groups and Network ACLs also allow the traffic.

10. CIDR Block of the VPC

The VPC uses the following IPv4 CIDR block:

172.31.0.0/16

The /16 prefix means that the first 16 bits identify the network and the remaining 16 bits are available for addressing inside the VPC. 

2^(32 - 16) = 65,536 IPv4 addresses

Therefore, the default VPC has an address range from: 

172.31.0.0 to 172.31.255.255

11. CIDR Blocks of the Subnets

The inspected subnets use /20 CIDR blocks:

172.31.16.0/20
172.31.32.0/20

A /20 subnet leaves 12 bits for host addressing. 

2^(32 - 20) = 4,096 IPv4 addresses

AWS reserves 5 IP addresses in each subnet. Therefore, each /20 subnet provides: 

4,096 - 5 = 4,091 usable IPv4 addresses

12. Public Subnet vs Private Subnet

A subnet is not public simply because an EC2 instance has a public IP address. A subnet is public when its associated route table has a route to an Internet Gateway.

Public Subnet

0.0.0.0/0 → Internet Gateway

Resources in a public subnet can communicate directly with the Internet if they have a public IP address and the security rules allow it.

Private Subnet

No direct route to an Internet Gateway

Resources in a private subnet do not receive direct inbound traffic from the Internet. They are typically accessed through internal services, load balancers, VPN, Direct Connect, bastion hosts or AWS Systems Manager Session Manager.

14. Production-Oriented Architecture

In a more professional architecture, the load balancer usually remains in public subnets, while EC2 instances are placed in private subnets. 

Internet
   │
   ▼
Internet Gateway
   │
   ▼
Public Subnets
   │
   ▼
Application Load Balancer
   │
   ▼
Private Subnets
   │
   ▼
Auto Scaling Group
   │
   ▼
EC2 Instances without public IP addresses

This reduces the attack surface because the EC2 instances are not directly reachable from the Internet.

15. Why Session Manager Matters

The current environment already follows an important security best practice: administration is performed through AWS Systems Manager Session Manager instead of SSH.

This provides several benefits:

  • No SSH key management is required.
  • Port 22 does not need to be open.
  • Access can be controlled through IAM.
  • Sessions can be logged and audited.
  • It is suitable for EC2 instances in private subnets.

16. NAT Gateway Concept

If EC2 instances are placed in private subnets, they cannot access the Internet directly through an Internet Gateway. For outbound access, such as installing packages or cloning a GitHub repository, a NAT Gateway is commonly used. 

Private EC2
   │
   ▼
Private Route Table
   │
   ▼
NAT Gateway
   │
   ▼
Internet Gateway
   │
   ▼
Internet

However, NAT Gateway has an hourly cost and data processing cost. For this reason, it was not created in this lab.

17. Default VPC vs Custom VPC

Default VPC Custom VPC
Created automatically by AWS Created explicitly by the user or by Infrastructure as Code
Good for initial learning and quick testing Recommended for controlled environments
Public subnets are usually available by default Public and private subnets are designed intentionally
Less control over network design Full control over CIDR, subnets and routing
Useful for simple labs Better for production-style architecture

18. Why No Custom VPC Was Created Yet

This lab focused on understanding the current network before creating a new one. Creating a VPC, subnets, route tables, Internet Gateway and associations in the same lab would introduce too many concepts at once.

The next lab will build a custom VPC using CloudFormation. This follows the same incremental approach used throughout the learning path.

19. AWS Best Practices Learned

  • Use multiple Availability Zones for high availability.
  • Place internet-facing load balancers in public subnets.
  • Place application servers in private subnets whenever possible.
  • Use route tables to control subnet behavior.
  • Use Security Groups to control allowed traffic.
  • Use Systems Manager Session Manager instead of SSH when possible.
  • Use Infrastructure as Code to create repeatable network environments.
  • Avoid unnecessary NAT Gateway usage in cost-sensitive labs.
