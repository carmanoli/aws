Lab 20 - AWS Systems Manager Run Command

1. Lab Objective

The objective of this lab is to introduce AWS Systems Manager as a practical tool for managing EC2 instances without using SSH. The lab focuses on registering an EC2 instance as a managed node and using Run Command to execute operational commands remotely.

2. Scenario

A cloud administrator needs to manage an EC2 instance securely without opening port 22, using SSH keys, or connecting directly to the server. The administrator must be able to verify system state and restart services remotely.

4. New AWS Concept Introduced

The new concept introduced in this lab is AWS Systems Manager, specifically:

  • Managed Nodes
  • SSM Agent
  • IAM Role for EC2
  • Run Command

5. Why This Matters in Real Environments

In professional AWS environments, administrators often avoid direct SSH access to production servers. Instead, they use AWS Systems Manager to execute commands, manage instances, collect inventory, apply patches, and troubleshoot systems securely.

This approach reduces the attack surface because it avoids exposing SSH to the internet and removes the need to manage SSH keys manually.

6. Systems Manager Requirements

For an EC2 instance to appear in Systems Manager as a managed node, the following requirements must be met:

  • The SSM Agent must be installed and running on the instance.
  • The EC2 instance must have outbound connectivity to Systems Manager endpoints.
  • The EC2 instance must have an IAM Role with the required permissions.
  • The instance must have time to register with Systems Manager.

7. Step 1 - Verify the EC2 Instance State

Open the EC2 console and verify that the instance lab-web-server is in the Running state.

8. Step 2 - Check Fleet Manager

Open AWS Systems Manager and navigate to:

Systems Manager → Fleet Manager

Initially, the instance did not appear. This indicated that one of the Systems Manager requirements was missing.

9. Step 3 - Check the EC2 IAM Role

In the EC2 console, select the instance lab-web-server and open the Security tab.

The instance did not have an IAM Role attached.

10. Step 4 - Create an IAM Role for Systems Manager

Open the IAM console and create a new role:

IAM → Roles → Create role

Configuration:

SettingValue
Trusted entityAWS service
Use caseEC2
PolicyAmazonSSMManagedInstanceCore
Role namelab-ec2-ssm-role

Technical note: In AWS, EC2 instances should use IAM Roles to obtain temporary credentials. Access keys should not be stored inside EC2 instances.

11. Step 5 - Attach the IAM Role to the EC2 Instance

Open the EC2 console and select lab-web-server.

Actions → Security → Modify IAM role

Select the role:

lab-ec2-ssm-role

Then click Update IAM role.

12. Step 6 - Wait for Systems Manager Registration

After attaching the IAM Role, wait a few minutes and refresh Fleet Manager.

The instance may take between 1 and 5 minutes to appear because Systems Manager registration is not always immediate.

13. Step 7 - Verify the SSM Agent

On the EC2 instance, verify that the SSM Agent is running:

sudo systemctl status amazon-ssm-agent

The expected result is:

Active: active (running)

14. Step 8 - Verify Connectivity to the SSM Endpoint

Run the following command from the EC2 instance:

curl -I https://ssm.eu-north-1.amazonaws.com

The result was:

HTTP/1.1 400 Bad Request

This response is acceptable for this test. The important point is that the instance successfully reached the Systems Manager endpoint over HTTPS.

15. Step 9 - Check the SSM Agent Logs

The SSM Agent logs were checked using:

sudo tail -50 /var/log/amazon/ssm/amazon-ssm-agent.log

The logs confirmed successful communication with Systems Manager:

Successfully opened websocket connection
Set up control channel successfully
Agent will take identity from EC2
Successfully created ssm-user

This confirmed that the EC2 instance was correctly registered and able to communicate with Systems Manager.

16. Step 10 - Run the First Remote Command

Open:

Systems Manager → Run Command → Run command

Select the command document:

AWS-RunShellScript

In the commands field, enter:

hostname

Target selection:

Choose instances manually → lab-web-server

Run the command.

17. Command Result

The command completed successfully and returned:

ip-172-31-31-235.eu-north-1.compute.internal

This confirms that the command was executed remotely on the EC2 instance through Systems Manager.

18. Step 11 - Check Apache Status Remotely

Using Run Command again, execute:

systemctl status httpd --no-pager

The result confirmed that Apache was running:

Active: active (running)

The service was also confirmed to be listening on port 80.

19. Step 12 - Restart Apache Remotely

To simulate a real operational task, Apache was restarted remotely using Run Command:

sudo systemctl restart httpd
sudo systemctl status httpd --no-pager

The result confirmed that Apache restarted successfully:

Active: active (running)

The new process ID confirmed that the service was actually restarted.

21. Security Benefits

  • No SSH connection was required.
  • No port 22 exposure was required.
  • No PEM key was needed.
  • No access keys were stored on the instance.
  • Authentication was handled through the EC2 IAM Role.
  • Commands were executed and audited through AWS Systems Manager.

23. Cost Considerations

AWS Systems Manager Run Command and Session Manager can generally be used without additional cost for standard EC2 management tasks. This lab reused the existing EC2 instance and did not require provisioning expensive resources.

To reduce costs when not working on the labs, stop the EC2 instance if it is not needed. Do not terminate it unless the infrastructure is no longer required.

25. Final Result

At the end of this lab, the EC2 instance lab-web-server was successfully managed through AWS Systems Manager. Remote commands were executed without SSH, and Apache was verified and restarted through Run Command.

26. Key Takeaways

  • Systems Manager is a practical service for managing EC2 instances securely.
  • IAM Roles are the correct way to grant AWS permissions to EC2 instances.
  • Run Command allows administrators to execute commands remotely without SSH.
  • SSM Agent communicates with AWS using HTTPS and WebSockets.
  • This approach is commonly used in enterprise environments to reduce operational risk.
