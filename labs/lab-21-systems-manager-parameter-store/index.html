Lab 21 - AWS Systems Manager Parameter Store

1. Lab Overview

In this lab, we used AWS Systems Manager Parameter Store to centralize application configuration and consume it from an EC2 instance without using SSH. The lab reused the existing EC2 instance that was already registered as a Systems Manager Managed Node in the previous lab.

The main goal was to understand how configuration values can be stored outside application code and retrieved dynamically by scripts, services, or applications.

New concept introduced: AWS Systems Manager Parameter Store.

2. Business Scenario

In a real enterprise environment, applications often require configuration values such as environment names, log levels, API URLs, database endpoints, bucket names, and feature flags.

Hardcoding these values directly in scripts or application code creates operational problems. Every time a configuration changes, teams may need to modify code, redeploy applications, or update multiple servers manually.

Parameter Store solves this problem by providing a centralized location for configuration values. Applications and automation scripts can retrieve these values at runtime.

3. Learning Objectives

  • Create a standard parameter in AWS Systems Manager Parameter Store.
  • Use hierarchical parameter naming.
  • Retrieve parameter values using the AWS CLI.
  • Use Systems Manager Run Command to execute commands on EC2 without SSH.
  • Use a parameter value inside a Bash script.
  • Update configuration centrally without changing the script.
  • Generate an Apache HTML page dynamically using a value from Parameter Store.

7. Why Parameter Store Matters

Before using Parameter Store, scripts often contain fixed values:

ENVIRONMENT="production"
LOG_LEVEL="INFO"
API_URL="https://api.example.com"

This approach is simple, but it does not scale well. If the value changes, the script must be edited and redeployed.

With Parameter Store, the script retrieves the value dynamically:

ENVIRONMENT=$(aws ssm get-parameter \
    --name "/lab/web/environment" \
    --region eu-north-1 \
    --query "Parameter.Value" \
    --output text)

The script remains unchanged, while the configuration can be updated centrally.

8. Step 1 - Create a Parameter

Open the AWS Console and navigate to:

Systems Manager > Parameter Store > Create parameter

Use the following settings:

FieldValue
Name/lab/web/environment
DescriptionEnvironment name for the lab web server
TierStandard
TypeString
Data typetext
Valueproduction

Click Create parameter.

9. Step 2 - Verify the Parameter

In Parameter Store, open the parameter /lab/web/environment and confirm:

  • Name: /lab/web/environment
  • Type: String
  • Tier: Standard
  • Value: production

10. Step 3 - Read the Parameter from EC2 Using Run Command

Navigate to:

Systems Manager > Run Command > Run command

Select the document:

AWS-RunShellScript

Select the target instance:

lab-web-server

Run the following command:

aws ssm get-parameter \
    --name "/lab/web/environment" \
    --region eu-north-1

Expected output:

{
    "Parameter": {
        "Name": "/lab/web/environment",
        "Type": "String",
        "Value": "production",
        "Version": 1,
        "ARN": "arn:aws:ssm:eu-north-1:ACCOUNT_ID:parameter/lab/web/environment",
        "DataType": "text"
    }
}
This confirmed that the EC2 instance could retrieve a value from Parameter Store using its IAM role and the AWS CLI.

11. Step 4 - Retrieve Only the Parameter Value

To retrieve only the value, use the AWS CLI --query and --output text options.

aws ssm get-parameter \
    --name "/lab/web/environment" \
    --region eu-north-1 \
    --query "Parameter.Value" \
    --output text

Expected output:

production

12. Step 5 - Use the Parameter in a Bash Script

The next step was to store the parameter value inside a Bash variable.

ENVIRONMENT=$(aws ssm get-parameter \
    --name "/lab/web/environment" \
    --region eu-north-1 \
    --query "Parameter.Value" \
    --output text)

echo "Current environment: $ENVIRONMENT"

Expected output:

Current environment: production

This is a common automation pattern. A script retrieves configuration from Parameter Store and then uses that value to control its behavior.

13. Step 6 - Update the Parameter Centrally

In Parameter Store, edit the parameter /lab/web/environment and change the value from:

production

to:

staging

Run the same script again:

ENVIRONMENT=$(aws ssm get-parameter \
    --name "/lab/web/environment" \
    --region eu-north-1 \
    --query "Parameter.Value" \
    --output text)

echo "Current environment: $ENVIRONMENT"

Expected output:

Current environment: staging
The script did not change, but the output changed because the configuration was updated centrally in Parameter Store.

14. Step 7 - Generate a Dynamic Apache Web Page

The parameter value was then used to generate an HTML page served by Apache.

ENVIRONMENT=$(aws ssm get-parameter \
    --name "/lab/web/environment" \
    --region eu-north-1 \
    --query "Parameter.Value" \
    --output text)

sudo tee /var/www/html/parameter-store.html > /dev/null <<'HTML_PAGE'
<!DOCTYPE html>
<html>
<head>
    <title>Lab 21 - Parameter Store</title>
</head>
<body>
    <h1>Lab 21 - AWS Systems Manager Parameter Store</h1>
    <p>Current environment: $ENVIRONMENT</p>
</body>
</html>
HTML_PAGE

sudo systemctl restart httpd

The page was accessed through the Application Load Balancer:

http://lab-web-alb-1301999270.eu-north-1.elb.amazonaws.com/parameter-store.html

Expected page content:

Lab 21 - AWS Systems Manager Parameter Store
Current environment: staging

15. Step 8 - Change the Parameter Back to Production

The parameter value was changed again from:

staging

back to:

production

The same Run Command script was executed again to regenerate the Apache page.

Expected result in the browser:

Current environment: production
This validated that configuration can be changed centrally without modifying the script, the EC2 instance, or the Apache configuration manually.

16. Final Validated Flow

Parameter Store /lab/web/environment = production | v Run Command executes script on EC2 | v EC2 retrieves parameter using AWS CLI | v Script writes /var/www/html/parameter-store.html | v Apache serves the page through the ALB

17. Troubleshooting

Problem: AccessDeniedException when reading the parameter

Possible cause:

  • The EC2 IAM role does not have permission to read parameters from Systems Manager Parameter Store.

Check:

aws sts get-caller-identity

Confirm that the EC2 instance is using the expected IAM role.

Problem: ParameterNotFound

Possible causes:

  • The parameter name is misspelled.
  • The wrong AWS Region is being used.

Check that the parameter name is exactly:

/lab/web/environment

Also confirm that the command uses:

--region eu-north-1

Problem: Run Command does not show the EC2 instance as a target

Possible causes:

  • The SSM Agent is not running.
  • The IAM role is not attached to the instance.
  • The instance has not yet registered as a Managed Node.

Check the managed node inventory in:

Systems Manager > Fleet Manager

Problem: The browser does not show the updated page

Possible causes:

  • The page was not regenerated after changing the parameter.
  • Apache was not restarted.
  • The browser cached the previous version.

Try refreshing the page or opening it in a private browsing window.

18. Best Practices

  • Use hierarchical names such as /application/component/setting.
  • Separate configuration from application code.
  • Use IAM policies to restrict access to specific parameter paths.
  • Use SecureString for sensitive values when appropriate.
  • Do not store passwords or secrets as plain String parameters.
  • Use Parameter Store for configuration values and AWS Secrets Manager for advanced secret lifecycle management.
  • Use descriptive parameter names that make the purpose clear.
  • Avoid hardcoding environment-specific values inside scripts.

19. Parameter Store vs Secrets Manager

FeatureParameter StoreSecrets Manager
Main use caseConfiguration values and simple secretsSecrets with rotation and lifecycle management
Common examplesEnvironment name, log level, API endpoint, bucket nameDatabase password, API key, OAuth token
Secret rotationManual or custom automationBuilt-in rotation support
CostStandard parameters are commonly used for low-cost configuration storageTypically used for managed secrets and may have additional cost

21. Key Takeaways

  • Parameter Store centralizes configuration values.
  • Hierarchical names help organize parameters clearly.
  • EC2 instances can retrieve parameters using the AWS CLI.
  • Run Command allows remote execution without SSH.
  • Scripts can consume parameters dynamically.
  • Configuration can be changed centrally without modifying the script.
  • This pattern is widely used in real cloud environments.
