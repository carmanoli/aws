4. What Are Nested Stacks?

A Nested Stack is a CloudFormation stack created and managed by another CloudFormation stack. The main stack is called the parent stack, and the stack created by it is called the child stack.

The parent stack uses the resource type:

AWS::CloudFormation::Stack

This allows a large infrastructure template to be divided into smaller, reusable templates.

5. Nested Stacks vs Cross-Stack References

Feature Cross-Stack References Nested Stacks
Relationship Independent stacks sharing values Child stacks controlled by a parent stack
Main mechanism Export and ImportValue AWS::CloudFormation::Stack
Lifecycle Stacks are managed independently Child stack lifecycle is managed by the parent stack
Use case Sharing values between separate stacks Modularizing a large CloudFormation project

8. Step 2 - Create the Child Template

The following file was created locally:

lab30-child-template.yaml

Initial child template:

AWSTemplateFormatVersion: '2010-09-09'
Description: Lab 30 - Child template for CloudFormation Nested Stack

Parameters:
  EnvironmentName:
    Type: String
    Default: production
    Description: Environment name used by the nested stack

Resources:
  LabNestedParameter:
    Type: AWS::SSM::Parameter
    Properties:
      Name: /lab/nested/environment
      Type: String
      Value: !Ref EnvironmentName
      Description: Parameter created by a nested CloudFormation stack

Outputs:
  NestedParameterName:
    Description: Name of the SSM Parameter created by the nested stack
    Value: !Ref LabNestedParameter

This template creates a lightweight Systems Manager Parameter Store parameter:

/lab/nested/environment

The file was uploaded to the S3 bucket.

9. Step 3 - Create the Parent Template

The following file was created locally:

lab30-parent-template.yaml

Parent template:

AWSTemplateFormatVersion: '2010-09-09'
Description: Lab 30 - Parent template for CloudFormation Nested Stack

Parameters:
  EnvironmentName:
    Type: String
    Default: production
    Description: Environment name passed from the parent stack to the child stack

Resources:
  LabNestedStack:
    Type: AWS::CloudFormation::Stack
    Properties:
      TemplateURL: https://lab-cfn-nested-templates-122445003699.s3.eu-north-1.amazonaws.com/lab30-child-template.yaml
      Parameters:
        EnvironmentName: !Ref EnvironmentName

Outputs:
  ChildStackId:
    Description: ID of the nested child stack
    Value: !Ref LabNestedStack

  ChildParameterName:
    Description: Name of the SSM Parameter created by the child stack
    Value: !GetAtt LabNestedStack.Outputs.NestedParameterName

The key element in this template is:

Type: AWS::CloudFormation::Stack

This tells CloudFormation to create another stack as a resource of the parent stack.

10. Step 4 - Create the Parent Stack

A new CloudFormation stack was created using the parent template:

lab30-nested-stack

During stack creation, CloudFormation required the following acknowledgement:

  • CAPABILITY_AUTO_EXPAND

This is expected because CloudFormation needs permission to expand and process the nested template.

After creation, the parent stack reached:

CREATE_COMPLETE

The nested child stack was also created automatically and reached:

CREATE_COMPLETE

11. Step 5 - Inspect the Nested Stack

The nested stack was visible from the parent stack resources tab as:

LabNestedStack

By opening this resource, it was possible to inspect the child stack created by the parent stack.

This confirmed that, from the parent stack perspective, the child stack is managed as a CloudFormation resource.

12. Step 6 - Update the Child Template

The child template was updated to include an additional output:

AWSTemplateFormatVersion: '2010-09-09'
Description: Lab 30 - Child template for CloudFormation Nested Stack

Parameters:
  EnvironmentName:
    Type: String
    Default: production
    Description: Environment name used by the nested stack

Resources:
  LabNestedParameter:
    Type: AWS::SSM::Parameter
    Properties:
      Name: /lab/nested/environment
      Type: String
      Value: !Ref EnvironmentName
      Description: Parameter created by a nested CloudFormation stack

Outputs:
  NestedParameterName:
    Description: Name of the SSM Parameter created by the nested stack
    Value: !Ref LabNestedParameter

  EnvironmentValue:
    Description: Current environment value
    Value: !Ref EnvironmentName

The updated file was uploaded to the same S3 object, replacing the previous version:

lab30-child-template.yaml

13. Step 7 - Update the Parent Stack

The parent stack was updated using the same parent template file:

lab30-parent-template.yaml

Although the parent template did not change, the TemplateURL still pointed to the child template in S3. CloudFormation reprocessed the child template and updated the nested stack.

The expected update flow was:

lab30-nested-stack  UPDATE_IN_PROGRESS
LabNestedStack      UPDATE_IN_PROGRESS
LabNestedStack      UPDATE_COMPLETE
lab30-nested-stack  UPDATE_COMPLETE

The child stack outputs were then updated and included:

  • NestedParameterName
  • EnvironmentValue
This validated that CloudFormation can update a nested stack through a parent stack update without recreating the entire infrastructure.

16. Key Takeaways

  • Nested Stacks are used to split large CloudFormation templates into smaller modules.
  • The parent stack manages the lifecycle of the child stacks.
  • The child template is usually stored in S3 and referenced through TemplateURL.
  • Parameters can be passed from the parent stack to the child stack.
  • Outputs from the child stack can be accessed from the parent stack using !GetAtt NestedStack.Outputs.OutputName.
  • Nested Stacks are different from Cross-Stack References because they are not independent stacks.
  • Nested Stacks are useful in enterprise environments to organize infrastructure by domain, such as network, security, compute, monitoring, and storage.
