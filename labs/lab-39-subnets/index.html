Lab 39 - Private Subnets

1. Lab Objective

The objective of this lab was to introduce private subnets inside the custom VPC created in Lab 38.

The existing production infrastructure remained untouched and continued to run inside the Default VPC. This lab only updated the networking stack lab38-custom-vpc-stack.

2. Architecture Before the Lab

  • Custom VPC: lab-custom-vpc
  • VPC CIDR block: 10.0.0.0/16
  • Public subnet A: 10.0.1.0/24
  • Public subnet B: 10.0.2.0/24
  • Internet Gateway: lab-custom-igw
  • Public Route Table: lab-public-route-table

3. New Resources Created

Resource Type Purpose
lab-private-subnet-a AWS::EC2::Subnet Private subnet in Availability Zone A
lab-private-subnet-b AWS::EC2::Subnet Private subnet in Availability Zone B
lab-private-route-table AWS::EC2::RouteTable Dedicated route table for private subnets
Private subnet route table associations AWS::EC2::SubnetRouteTableAssociation Explicitly associate both private subnets with the private route table

4. Technical Design Decisions

4.1 CIDR Blocks

The private subnets were created with separate CIDR blocks:

  • 10.0.11.0/24 for lab-private-subnet-a
  • 10.0.12.0/24 for lab-private-subnet-b

These ranges are inside the main VPC CIDR block 10.0.0.0/16 and do not overlap with the public subnets.

4.2 Availability Zones

The private subnets were placed in the same two Availability Zones as the public subnets. This prepares the architecture for high availability, where public and private tiers can be distributed across multiple Availability Zones.

4.3 Public IP Assignment

The private subnets were configured with:

MapPublicIpOnLaunch: false

This prevents EC2 instances launched in these subnets from automatically receiving public IPv4 addresses.

4.4 Private Route Table

A dedicated private route table was created, but no default route to the Internet Gateway was added.

The private route table only contains the automatic local VPC route:

10.0.0.0/16 → local

This allows communication inside the VPC, but prevents direct Internet access.

Important: A subnet is considered public or private based mainly on its route table. A public subnet has a route to an Internet Gateway. A private subnet does not.

5. CloudFormation Template Additions

5.1 Private Subnets

LabPrivateSubnetA:
  Type: AWS::EC2::Subnet
  Properties:
    VpcId: !Ref LabCustomVPC
    CidrBlock: 10.0.11.0/24
    AvailabilityZone: !Select [0, !GetAZs '']
    MapPublicIpOnLaunch: false
    Tags:
      - Key: Name
        Value: lab-private-subnet-a

LabPrivateSubnetB:
  Type: AWS::EC2::Subnet
  Properties:
    VpcId: !Ref LabCustomVPC
    CidrBlock: 10.0.12.0/24
    AvailabilityZone: !Select [1, !GetAZs '']
    MapPublicIpOnLaunch: false
    Tags:
      - Key: Name
        Value: lab-private-subnet-b

5.2 Private Route Table

LabPrivateRouteTable:
  Type: AWS::EC2::RouteTable
  Properties:
    VpcId: !Ref LabCustomVPC
    Tags:
      - Key: Name
        Value: lab-private-route-table

5.3 Route Table Associations

LabPrivateSubnetARouteTableAssociation:
  Type: AWS::EC2::SubnetRouteTableAssociation
  Properties:
    SubnetId: !Ref LabPrivateSubnetA
    RouteTableId: !Ref LabPrivateRouteTable

LabPrivateSubnetBRouteTableAssociation:
  Type: AWS::EC2::SubnetRouteTableAssociation
  Properties:
    SubnetId: !Ref LabPrivateSubnetB
    RouteTableId: !Ref LabPrivateRouteTable

9. Enterprise Relevance

This architecture is common in enterprise AWS environments. Public subnets are typically used for internet-facing components, such as Application Load Balancers. Private subnets are used for internal workloads, such as application servers, backend services, containers, databases, and internal APIs.

This reduces the attack surface because backend resources do not have direct inbound exposure from the Internet.

11. Cost Considerations

This lab created only VPC networking resources that do not generate significant direct cost. No NAT Gateway, EC2 instance, RDS database, or Load Balancer was created in this lab.

This keeps the lab aligned with cost optimization and Free Tier awareness.

12. Lab Conclusion

Lab 39 was completed successfully. The custom VPC now contains two public subnets and two private subnets distributed across two Availability Zones. The private subnets are associated with a dedicated private route table and do not have direct Internet access.
