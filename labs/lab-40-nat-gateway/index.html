Lab 40 - Amazon VPC NAT Gateway using CloudFormation

Objective

In this laboratory we introduced the Amazon VPC NAT Gateway service and learned how private subnets can securely access the Internet without exposing their instances to inbound connections.

The networking infrastructure continued to evolve using the existing CloudFormation networking stack created in previous laboratories.

This meant that instances deployed inside private subnets had no Internet access.

Why is a NAT Gateway Required?

Private subnets intentionally have no route to the Internet Gateway. This protects backend systems from direct Internet access.

However, many workloads still need outbound connectivity, for example:

  • Operating System updates
  • Downloading application packages
  • Connecting to AWS APIs
  • Retrieving software from GitHub
  • Accessing external services

A NAT Gateway provides outbound Internet connectivity while blocking inbound connections initiated from the Internet.

Internet Gateway vs NAT Gateway

No hourly charge Hourly and data processing charges
Internet-facing resources Backend infrastructure

Enterprise Best Practices

  • Deploy one NAT Gateway per Availability Zone.
  • Associate each private subnet with a NAT Gateway in the same AZ.
  • Avoid cross-AZ traffic charges.
  • Use Infrastructure as Code for all networking resources.
  • Validate every deployment using Change Sets.
  • Remove temporary resources to reduce costs.

Cost Optimization

Unlike Internet Gateways, NAT Gateways incur hourly charges and data processing charges.

After validating the laboratory, the following resources were removed through CloudFormation:

  • LabNatGateway
  • LabNatGatewayEIP
  • LabPrivateDefaultRoute

This kept the CloudFormation stack consistent while eliminating unnecessary costs.
