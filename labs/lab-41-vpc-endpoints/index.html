Lab 41 – Amazon VPC Gateway Endpoint for Amazon S3

This lab introduces one of the most important networking concepts in AWS: accessing AWS managed services privately without using the public Internet.

Instead of routing traffic through a NAT Gateway, Amazon S3 can be reached using a VPC Gateway Endpoint, improving security while eliminating NAT Gateway costs for S3 traffic.

Objectives

  • Understand Amazon VPC Endpoints
  • Understand Gateway Endpoints
  • Create an Amazon S3 Gateway Endpoint using CloudFormation
  • Associate the endpoint with the Private Route Table

  • Restrict access using an Endpoint Policy

Why VPC Endpoints?

Private instances frequently need to access AWS services such as Amazon S3. Without a VPC Endpoint, traffic would normally require a NAT Gateway to reach the public S3 endpoint.

A Gateway Endpoint provides a private connection between the VPC and Amazon S3 without using the Internet.

S3 Bucket

Type: AWS::S3::Bucket

Automatic Route Table Update

The Gateway Endpoint automatically inserted a route into the Private Route Table. No AWS::EC2::Route resource was required.

Destination Target
pl-xxxxxxxx vpce-xxxxxxxx
10.0.0.0/16 local

The destination beginning with pl- is an AWS Managed Prefix List representing Amazon S3 for the selected Region.

Endpoint Policy

Initially, the endpoint allowed access to every S3 bucket. For better security the policy was updated to allow access only to the bucket created during the lab. 

Allow:
- ListBucket

Bucket:
arn:aws:s3:::lab-s3-endpoint-ACCOUNT-REGION

Allow:
- GetObject
- PutObject
- DeleteObject

Objects:
arn:aws:s3:::lab-s3-endpoint-ACCOUNT-REGION/*

Security Benefits

  • No Internet Gateway required
  • No public IP addresses required
  • No NAT Gateway required for S3 traffic
  • Traffic never leaves the AWS network
  • Can be combined with Bucket Policies
  • Can be combined with IAM Policies
  • Supports least privilege access

For workloads that only require Amazon S3 access, Gateway Endpoints provide a significant cost reduction.

Enterprise Best Practices

  • Use Gateway Endpoints whenever workloads require only S3 or DynamoDB.
  • Restrict access using Endpoint Policies.
  • Combine Endpoint Policies with IAM and Bucket Policies.
  • Reduce NAT Gateway usage to minimize operational costs.
  • Deploy Interface Endpoints only for services that require them.

AWS Well-Architected Framework

  • Security: private connectivity to AWS services.
  • Cost Optimization: eliminate unnecessary NAT Gateway traffic.
  • Reliability: AWS-managed highly available endpoint.
  • Operational Excellence: infrastructure managed using CloudFormation.

CloudFormation Change Set

The update introduced only two changes:

  • Modify AWS::EC2::VPCEndpoint (No Replacement)

The production workload remained unaffected throughout the deployment.

Conclusion

In this lab we implemented a production-style Amazon S3 Gateway Endpoint using CloudFormation. The endpoint automatically updated the Private Route Table, allowing resources in private subnets to access Amazon S3 without requiring a NAT Gateway or Internet access. We also applied a restrictive Endpoint Policy, demonstrating how enterprises implement least-privilege access while reducing network costs and improving security.
