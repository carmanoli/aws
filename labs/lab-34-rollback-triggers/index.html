Lab 34 - CloudFormation Rollback Triggers

1. Lab Objective

The objective of this lab is to understand and validate AWS CloudFormation Rollback Triggers using Amazon CloudWatch Alarms.

Rollback Triggers allow CloudFormation to monitor one or more CloudWatch Alarms during a stack creation or update. If any configured alarm enters the ALARM state, CloudFormation automatically rolls back the operation.

This lab continues from the previous CloudFormation labs. The existing infrastructure is managed mainly through the stack:

4. Technical Decision

At first, a CloudWatch Alarm based on EC2 CPU utilization was considered. However, CPU-based alarms are not reliable for this lab because the EC2 instance may remain almost idle and never cross the threshold during the CloudFormation update window.

For this reason, a custom CloudWatch metric was used. This allows the alarm state to be controlled manually and makes the rollback test deterministic.

5. Initial CloudWatch Alarm Attempt

A CPU-based alarm was created with the following configuration:

  • Metric: CPUUtilization
  • Statistic: Average
  • Period: 1 minute
  • Condition: greater than 95
  • Datapoints to alarm: 1 out of 1
  • Alarm name: lab-rollback-trigger-alarm
  • No SNS notification configured

This alarm was useful to understand the interface, but it was not ideal for the rollback simulation.

6. Important Console Detail

When adding a rollback trigger in the CloudFormation console, the alarm must be referenced by its full ARN, not only by its name.

Incorrect value:

lab-rollback-trigger-alarm

Correct value format:

arn:aws:cloudwatch:eu-north-1:122445003699:alarm:lab-rollback-trigger-alarm

If only the alarm name is used, CloudFormation returns the validation error:

ARNs must start with 'arn:'

7. Creating a Custom CloudWatch Metric

A custom metric was published using AWS CloudShell:

aws cloudwatch put-metric-data \
  --namespace Lab/Rollback \
  --metric-name DeploymentHealth \
  --value 0

The custom metric represents deployment health:

  • 0 means healthy
  • 1 means unhealthy

8. Creating the Controlled Rollback Alarm

A new CloudWatch Alarm was created based on the custom metric:

SettingValue
Alarm namelab-rollback-custom-alarm
NamespaceLab/Rollback
Metric nameDeploymentHealth
StatisticAverage
Period1 minute
ConditionGreater than 0
Datapoints to alarm1 out of 1
NotificationNone

The metric was published again with value 0 until the alarm entered the OK state.

aws cloudwatch put-metric-data \
  --namespace Lab/Rollback \
  --metric-name DeploymentHealth \
  --value 0

9. Configuring the Rollback Trigger

The CloudFormation stack lab26-asg-stack was updated using the current template. In the stack options page, the following rollback configuration was applied:

SettingValue
Rollback triggerARN of lab-rollback-custom-alarm
Monitoring time1 minute

The update completed successfully with:

UPDATE_COMPLETE

10. Simulating a Failed Deployment

To trigger a real CloudFormation update, the parameter DesiredCapacity was changed from:

1

to:

2

As soon as the stack entered UPDATE_IN_PROGRESS, the custom metric was changed to an unhealthy value:

aws cloudwatch put-metric-data \
  --namespace Lab/Rollback \
  --metric-name DeploymentHealth \
  --value 1

This caused the alarm lab-rollback-custom-alarm to enter the ALARM state.

11. Observed CloudFormation Rollback

The stack events showed that the update was automatically rolled back:

UPDATE_ROLLBACK_IN_PROGRESS
UPDATE_ROLLBACK_COMPLETE_CLEANUP_IN_PROGRESS
UPDATE_ROLLBACK_COMPLETE

The relevant event message was:

The following resource(s) failed to update: [LabWebAutoScalingGroup].

The Auto Scaling Group resource also showed:

Resource skipped during UpdateRollback

This confirms that the Rollback Trigger worked correctly. CloudFormation detected the alarm state and reverted the update automatically.

12. Restoring the Alarm to Healthy State

After the rollback test, the custom metric was restored to a healthy value:

aws cloudwatch put-metric-data \
  --namespace Lab/Rollback \
  --metric-name DeploymentHealth \
  --value 0

After one or two minutes, the alarm returned to:

OK

13. Stack Policies vs Rollback Triggers

Stack Policies Rollback Triggers
Protect resources from being updated. Monitor health during stack operations.
Act before a resource update is allowed. Act during or after an update.
Based on update permissions for resources. Based on CloudWatch Alarm state.
Prevent accidental changes to critical resources. Automatically roll back unhealthy deployments.
Useful for protecting databases, ASGs, IAM roles, and networking resources. Useful for CI/CD, application health checks, error rates, latency, and custom deployment metrics.

14. Real-World Use Cases

Rollback Triggers are useful in enterprise environments where infrastructure updates are automated and must be automatically reverted if health signals degrade.

Typical examples include:

  • Application error rate increases after deployment
  • Load balancer target health becomes unhealthy
  • Latency exceeds an acceptable threshold
  • CPU or memory utilization becomes abnormal
  • A custom deployment metric reports failure
  • A CI/CD pipeline publishes a custom health metric during deployment

16. Key Lessons Learned

  • Rollback Triggers are configured at stack operation level.
  • CloudFormation requires the full CloudWatch Alarm ARN.
  • Rollback Triggers do not prevent changes; they monitor health and roll back if necessary.
  • Custom CloudWatch metrics are useful for deterministic lab simulations.
  • Stack Policies and Rollback Triggers solve different problems and complement each other.
  • This feature is especially important in automated deployment pipelines.
