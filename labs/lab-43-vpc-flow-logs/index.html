Lab 43 - Amazon VPC Flow Logs

1. Lab Objective

The objective of this lab was to introduce Amazon VPC Flow Logs and use them to monitor network traffic inside the custom VPC created in previous labs.

The lab focused on capturing network metadata, understanding accepted and rejected traffic, sending Flow Logs to Amazon CloudWatch Logs, and analysing real traffic generated by an EC2 instance.

2. Context Before the Lab

Before this lab, the environment already contained a custom VPC with public and private subnets, an Internet Gateway, public and private route tables, a Gateway VPC Endpoint for Amazon S3, and previously tested Interface VPC Endpoints.

The production workload from earlier labs was still running in the Default VPC through the lab26-asg-stack. The custom VPC did not yet contain any EC2 instance, ALB, Auto Scaling Group, or active workload.

3. Why VPC Flow Logs Matter

Amazon VPC Flow Logs provide visibility into network traffic going to and from network interfaces inside a VPC. They are widely used in enterprise environments for troubleshooting, security monitoring, compliance, and forensic investigation.

Flow Logs do not capture packet payloads. They capture metadata such as source IP address, destination IP address, ports, protocol, number of packets, number of bytes, action, and log status.

5. CloudFormation Resources Added

The following resources were added to the existing lab38-custom-vpc-stack using a CloudFormation Change Set:

  • AWS::Logs::LogGroup for CloudWatch Logs
  • AWS::IAM::Role for VPC Flow Logs delivery permissions
  • AWS::EC2::FlowLog to enable Flow Logs at VPC level
  • AWS::EC2::SecurityGroup for the temporary EC2 instance
  • AWS::IAM::Role for Systems Manager access
  • AWS::IAM::InstanceProfile for the EC2 instance
  • AWS::EC2::Instance to generate real traffic

6. CloudWatch Log Group

The Flow Logs were configured to send records to the following CloudWatch Logs group: 

/aws/vpc/lab-custom-vpc-flow-logs

The retention period was set to one day to reduce costs during the lab.

7. IAM Role for VPC Flow Logs

VPC Flow Logs require permission to publish records to CloudWatch Logs. For that reason, an IAM Role was created with a trust relationship allowing the service vpc-flow-logs.amazonaws.com to assume the role.

The role included permissions such as: 

logs:CreateLogStream
logs:PutLogEvents
logs:DescribeLogGroups
logs:DescribeLogStreams

8. VPC Flow Log Configuration

The Flow Log was created at VPC level using the following logic:

Property Value Purpose
ResourceType VPC Capture traffic metadata for the entire VPC.
TrafficType ALL Capture both accepted and rejected traffic.
LogDestinationType cloud-watch-logs Send Flow Logs to CloudWatch Logs for easy analysis.

11. Flow Log Validation

After the EC2 instance started successfully, CloudWatch Logs began receiving VPC Flow Log records. This confirmed that the Flow Log was correctly attached to the custom VPC.

Example record: 

2 122445003699 eni-01b1540974260f99d 10.0.1.65 16.171.83.98 38104 443 6 4 232 1783609125 1783609151 ACCEPT OK

12. Flow Log Field Analysis

Field Example Meaning
version 2 Flow Log record version.
account-id 122445003699 AWS account ID.
interface-id eni-01b1540974260f99d Elastic Network Interface that generated the record.
srcaddr 10.0.1.65 Source IP address.
dstaddr 16.171.83.98 Destination IP address.
srcport 38104 Source port.
dstport 443 Destination port, in this case HTTPS.
protocol 6 Protocol number. 6 means TCP.
packets 4 Number of packets transferred.
bytes 232 Number of bytes transferred.
action ACCEPT The traffic was allowed.
log-status OK The log record was successfully delivered.

13. Accepted Traffic

Accepted traffic was observed between the EC2 instance and external AWS service endpoints over HTTPS. 

10.0.1.65 -> 16.171.83.98 destination port 443 ACCEPT

This indicates that the instance was allowed to communicate using TCP over port 443. This is commonly seen when the instance communicates with AWS services such as Systems Manager, package repositories, or other HTTPS endpoints.

14. Rejected Traffic

Rejected traffic was also observed from public Internet IP addresses attempting to reach the EC2 instance on unexpected ports. 

172.232.27.232 -> 10.0.1.65 destination port 40001 REJECT
78.128.114.110 -> 10.0.1.65 destination port 178 REJECT
51.159.110.167 -> 10.0.1.65 destination port 25565 REJECT

These records show that the instance, because it was temporarily deployed in a public subnet with a public IP address, was visible to unsolicited Internet traffic. The Security Group rejected traffic that was not explicitly allowed.

This was an important security observation. It demonstrated why enterprise workloads should normally run in private subnets behind a public Application Load Balancer.

15. What VPC Flow Logs Do Not Capture

VPC Flow Logs do not capture the payload or content of packets. For example, they do not show HTTP headers, request bodies, usernames, passwords, or files transferred over the network.

They also do not replace application logs, web server logs, CloudTrail, or CloudWatch metrics. They are a network metadata logging service.

16. Comparison with CloudTrail and CloudWatch

Service Main Purpose Example Question Answered
VPC Flow Logs Network traffic metadata Which IP tried to connect to this ENI and was it accepted or rejected?
AWS CloudTrail AWS API activity auditing Who created, modified, or deleted an AWS resource?
Amazon CloudWatch Metrics Performance and operational metrics Is CPU, network, or load balancer activity increasing?
Amazon CloudWatch Logs Centralized log storage and analysis What log events were produced by an application or AWS service?

17. Enterprise Use Cases

  • Troubleshooting blocked connectivity between subnets or services.
  • Identifying unexpected inbound traffic from the Internet.
  • Detecting unusual outbound traffic from EC2 instances.
  • Supporting incident response and forensic investigation.
  • Validating Security Group and Network ACL behaviour.
  • Providing evidence for compliance and auditing requirements.

18. Relation to AWS Solutions Architect Associate SAA-C03

VPC Flow Logs are relevant to the SAA-C03 exam because they relate to secure architectures, operational excellence, network troubleshooting, monitoring, and compliance.

A Solutions Architect should understand when to use Flow Logs, what they capture, what they do not capture, and how they differ from CloudTrail and CloudWatch metrics.

19. Cost Considerations

VPC Flow Logs can generate CloudWatch Logs ingestion and storage costs. To reduce cost in this lab, the log retention period was configured as one day.

The EC2 instance created in this lab should not be considered a final production architecture. It was created to generate traffic and validate the Flow Logs. The next labs should move the workload into private subnets and eventually remove duplicated resources from the older Default VPC stack.

22. Conclusion

In this lab, Amazon VPC Flow Logs were successfully enabled for the custom VPC using CloudFormation. Logs were delivered to CloudWatch Logs and real network traffic was analysed.

The lab demonstrated both accepted and rejected traffic, showing how Flow Logs can be used for troubleshooting, security monitoring, and forensic investigation.

The rejected inbound traffic also highlighted an important architectural lesson: EC2 instances should generally not be directly exposed in public subnets. The next labs should continue the migration toward a production-style architecture using an Application Load Balancer in public subnets and EC2 instances in private subnets.
