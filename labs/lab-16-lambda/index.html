6. Serverless Computing Concept

AWS Lambda is a serverless compute service that allows code to run without provisioning or managing servers. With Lambda, the cloud provider manages the infrastructure, operating system, scaling, availability, and execution environment.

In a traditional server-based model, you usually manage:

  • Servers
  • Operating systems
  • Runtime installation
  • Patching
  • Scaling
  • High availability

With AWS Lambda, you focus mainly on:

  • Writing code
  • Defining permissions
  • Configuring triggers
  • Monitoring executions
Lambda is commonly used in event-driven architectures, automation tasks, data processing, API backends, security workflows, and integration between AWS services.

Step 1 - Open AWS Lambda

  1. Open the AWS Management Console.
  2. Go to Services.
  3. Search for Lambda.
  4. Open the AWS Lambda service.

Step 2 - Create the Lambda Function

  1. Click Create function.
  2. Select Author from scratch.
  3. Configure the function:
  • Function name: lab-hello-lambda
  • Runtime: Python 3.13
  • Architecture: x86_64

In the permissions section, select:

Create a new role with basic Lambda permissions

This allows AWS to automatically create an IAM execution role with the minimum permissions required for the function to write logs to CloudWatch Logs.

Click Create function.

8. Initial Lambda Code

After the function is created, AWS provides a default Python code structure similar to this:

import json

def lambda_handler(event, context):
    return {
        'statusCode': 200,
        'body': json.dumps('Hello from Lambda!')
    }

9. Understanding the Lambda Handler

Handler

The handler is the function that AWS Lambda executes when the Lambda function is invoked.

In this lab, the handler is:

lambda_handler

The full handler reference is usually:

lambda_function.lambda_handler

This means:

  • lambda_function is the Python file name.
  • lambda_handler is the function name inside the file.

event

The event parameter contains the data passed to the Lambda function during invocation.

Examples of future event sources include:

  • Amazon SNS messages
  • Amazon S3 object upload events
  • Amazon EventBridge scheduled events
  • API Gateway HTTP requests
  • DynamoDB Streams records

context

The context parameter contains runtime information about the current invocation.

It can include information such as:

  • Function name
  • Request ID
  • Remaining execution time
  • Memory configuration
  • Log stream name

12. Cold Start and Warm Start

Cold Start

A cold start happens when Lambda needs to create a new execution environment before running the function. This can add a small amount of latency to the first invocation.

Warm Start

A warm start happens when Lambda reuses an existing execution environment from a previous invocation. This usually makes the function start faster.

13. Updating the Function Code

Replace the default function code with the following:

import json

def lambda_handler(event, context):

    name = event.get("name", "World")

    return {
        "statusCode": 200,
        "body": json.dumps(f"Hello {name}!")
    }

Click Deploy to save and deploy the updated code.

15. Understanding the Updated Code

name = event.get("name", "World")

This line reads the name key from the event object. If the key exists, its value is used. If the key does not exist, the default value World is used.

With this test event:

{
  "name": "Carlos"
}

The function returns:

Hello Carlos!

16. IAM Execution Role

The Lambda function does not use the IAM permissions of the user who created it. Instead, it uses its own IAM execution role.

In this lab, AWS automatically created a role with basic Lambda permissions. These permissions allow the function to write logs to CloudWatch Logs.

Typical basic permissions include:

  • Create log groups
  • Create log streams
  • Put log events
In professional environments, Lambda permissions should follow the principle of least privilege. A Lambda function should only receive the permissions required to perform its task.

17. CloudWatch Logs

Every Lambda invocation can generate logs in Amazon CloudWatch Logs. This is essential for troubleshooting, auditing, and monitoring serverless applications.

The log group created for this function follows this format:

/aws/lambda/lab-hello-lambda

Logs are useful to inspect:

  • Invocation start and end
  • Execution duration
  • Errors
  • Printed debug messages
  • Request IDs

18. Professional Best Practices

  • Use clear and consistent naming conventions, such as lab-....
  • Apply the principle of least privilege to Lambda execution roles.
  • Use CloudWatch Logs for troubleshooting and operational visibility.
  • Keep functions small and focused on a single responsibility.
  • Use environment variables instead of hardcoding configuration values.
  • Monitor errors, duration, throttles, and invocation count.
  • Use Infrastructure as Code for production deployments.
  • Avoid giving Lambda functions broad permissions such as full access policies.

19. Certification Notes for AWS SAA-C03

Important concepts for the AWS Solutions Architect Associate exam:

  • Lambda is a serverless compute service.
  • Lambda automatically scales based on the number of incoming events.
  • Lambda functions are invoked by events.
  • Each function uses an IAM execution role.
  • Lambda integrates natively with CloudWatch Logs.
  • Lambda is commonly used with SNS, S3, EventBridge, SQS, DynamoDB Streams, and API Gateway.
  • Lambda is suitable for short-running, event-driven workloads.
  • Lambda is not ideal for long-running processes that exceed the maximum execution duration.

20. Cost Considerations

AWS Lambda includes a generous free tier. This lab uses only manual test invocations and very small amounts of compute time. The expected cost for this lab is effectively zero under normal Free Tier usage.

CloudWatch Logs may store a small amount of log data. For this lab, the volume is minimal.

To avoid unnecessary costs in long-term environments, remove unused Lambda functions and CloudWatch log groups when they are no longer needed.
