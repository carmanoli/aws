Lab 23 – Amazon EC2 Auto Scaling Recovery

Objective

Learn how Amazon EC2 Auto Scaling automatically replaces failed EC2 instances, maintaining service availability without manual intervention.

During this lab the existing infrastructure built in previous labs is reused.

Prerequisites

  • Launch Template (lab-web-template)
  • Application Load Balancer
  • Target Group
  • Systems Manager
  • Session Manager
  • Parameter Store
  • GitHub automatic deployment through User Data
  • IAM Role for Systems Manager (lab-ec2-ssm-role)

Step 1 - Verify the Launch Template

The original Launch Template was created before Systems Manager was introduced. Therefore it did not contain the IAM Instance Profile required for Session Manager.

Navigate to: 

EC2
    Launch Templates
        lab-web-template
Verify that no IAM Instance Profile is configured.

Step 2 - Create Launch Template Version 2

Create a new Launch Template version.

Modify only:

Setting Value
IAM Instance Profile lab-ec2-ssm-role
Description Version 2 - Add SSM IAM Instance Profile
Always create a new Launch Template version instead of modifying an existing one. This is the recommended enterprise practice.

Step 6 - Validate the New Instance

After creation, Auto Scaling launches a new EC2 instance.

Validate:

  • Instance is Running
  • Target Group registers the instance automatically
  • Initial health status may be Unhealthy
  • Eventually becomes Healthy
It is normal for the Target Group to report Unhealthy during User Data execution. The instance becomes Healthy after Apache starts responding successfully.

Step 7 - Remove the Old Manual Instance

The original EC2 instance was created manually in previous labs and does not belong to the Auto Scaling Group.

Terminate the manual instance.

The Auto Scaling Group instance continues serving the application.

Step 8 - Test Automatic Recovery

Terminate the EC2 instance managed by the Auto Scaling Group.

Observe:

  1. Instance enters Shutting-down.
  2. Instance becomes Terminated.
  3. Auto Scaling detects desired capacity is no longer satisfied.
  4. A replacement instance is launched automatically.
  5. User Data executes.
  6. GitHub deployment runs automatically.
  7. Apache starts.
  8. Target Group registers the instance.
  9. ALB Health Checks succeed.
  10. Application becomes available again.

Enterprise Best Practices

  • Never modify existing Launch Templates directly.
  • Create new Launch Template versions.
  • Use Auto Scaling Groups even for a single production instance.
  • Treat EC2 instances as disposable resources.
  • Automate instance configuration using User Data.
  • Use Systems Manager instead of SSH whenever possible.
  • Always protect applications with Load Balancers.

Lab Outcome

The infrastructure successfully recovered from an intentional EC2 failure. A replacement instance was automatically created, configured through User Data, registered in the Target Group, became Healthy behind the Application Load Balancer, and was fully manageable using AWS Systems Manager Session Manager.
