Lab 10

Auto Scaling Group with an Application Load Balancer

Extend Lab 09 so AWS creates, replaces and removes EC2 instances automatically while an ALB distributes web traffic.

EC2Auto ScalingCloudWatchALB

Objectives and final architecture

Internet
   │
Application Load Balancer
   │
Target Group · HTTP :80
   │
Auto Scaling Group
   ├── EC2 · eu-north-1a
   └── EC2 · eu-north-1b

Minimum: 2 · Desired: 2 · Maximum: 4

This lab practises launch templates, user data, EC2 and ELB health checks, target tracking, CloudWatch alarms, instance warmup, scale out, scale in and connection draining.

1. Confirm the Lab 09 environment

  1. Confirm that both Lab 09 EC2 instances are running.
  2. Open EC2 → Load Balancers and confirm that alb-web-lab9 is Active.
  3. Open tg-web-lab9 and confirm both targets are Healthy.
  4. Open the ALB DNS name and refresh to verify that responses alternate between servers.

2. Create a launch template

A launch template describes how the Auto Scaling Group creates each new instance. Creating a template from an existing instance was not suitable because Apache had been installed manually and the copied user data was empty.

The EC2 security group must allow TCP port 80 from the ALB security group. If using Session Manager, also include the required IAM instance profile.

User data

#!/bin/bash
dnf update -y
dnf install -y httpd
systemctl enable httpd
systemctl start httpd

TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
INSTANCE_ID=$(curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id)

cat <<'HTML_EOF' > /var/www/html/index.html
<html>
<head><title>AWS Auto Scaling</title></head>
<body style="font-family:Arial;text-align:center;margin-top:60px">
<h1>AWS Auto Scaling Lab</h1>
<h2>Server created automatically</h2>
<p><b>Instance ID:</b> REPLACE_INSTANCE_ID</p>
</body>
</html>
HTML_EOF

sed -i "s/REPLACE_INSTANCE_ID/$INSTANCE_ID/g" /var/www/html/index.html

This installs and starts Apache, retrieves the instance ID securely through IMDSv2 and creates a page identifying the server.

3. Create the Auto Scaling Group

Open EC2 → Auto Scaling Groups → Create Auto Scaling group.

Important: The ASG subnets must match Availability Zones enabled on the ALB. An instance launched in eu-north-1c appeared as Unused because the ALB was enabled only in eu-north-1a and eu-north-1b.

4. Verify automatic instance creation

  1. Open ASG-WebServers → Instance management.
  2. Confirm two instances show InService and Healthy.
  3. Open the target group. During the transition it may contain two manually created instances and two ASG instances.
  4. Open the ALB DNS name and refresh. Pages from new instances display their instance IDs.

Normal startup states include Initial, Healthy, Unhealthy, Unused and Draining.

5. Fix an Unused target

The ASG was initially configured with eu-north-1c, while the ALB used only eu-north-1a and eu-north-1b. Edit the ASG network settings, remove the eu-north-1c subnet and retain only the ALB-enabled subnets. The ASG will replace the incorrectly placed instance.

6. Configure target tracking

Open ASG-WebServers → Automatic scaling → Create dynamic scaling policy:

SettingValue
Policy typeTarget tracking scaling policy
MetricAverage CPU utilization
Target value50%
Instance warmup120 seconds
Scale inEnabled

For testing, the target was temporarily changed to 30%. With one instance at 100% and another close to 0%, average CPU is roughly 50%, which may not trigger a 50% target.

7. Generate CPU load

Connect to one ASG-managed instance with Session Manager or SSH:

sudo dnf install -y stress-ng
cd /tmp
stress-ng --cpu 2 --timeout 600s

If stress-ng reports that its temporary path must be readable and writable, use:

stress-ng --temp-path /tmp --cpu 2 --timeout 600s

In a second session, run top and confirm that CPU usage is close to 100%.

8. Observe scale out

Open the ASG Activity history. Expected events include a target-tracking alarm, desired capacity changing from 2 to 3 and a new EC2 instance launching.

During this test the ASG briefly reached four instances because the target was low, load rose quickly and new instances were still in their warmup period. This is normal in an aggressive scaling test.

9. Observe scale in

Wait for the stress command to finish or stop it with Ctrl+C. When average CPU falls, the ASG gradually returns desired capacity to two. Instances may display Terminating, Draining or Waiting for ELB connection draining while existing connections finish.

10. Interpret the events

  • TargetTracking in ALARM: average CPU exceeded the configured target.
  • Launching a new EC2 instance: the ASG is adding capacity.
  • Health check failure: the ASG removed an unhealthy instance and launched a replacement.
  • Connection draining: the ALB is allowing active connections to finish before deregistration.

11. Final architecture

Once the ASG is working, remove the manually created Lab 09 instances from the target group and terminate them when no longer needed. A cleaner architecture contains only the ALB, target group, ASG and EC2 instances managed by the ASG.

Certification notes

  • The ALB sends traffic to healthy targets registered in its target group.
  • The launch template defines how new EC2 instances are created.
  • User data automates instance configuration.
  • The ASG maintains desired capacity and replaces unhealthy instances.
  • Target tracking keeps a selected metric close to a target value using CloudWatch alarms.
  • Instance warmup prevents new instances from influencing metrics too early.
  • The ASG and ALB should use aligned Availability Zones.

Cleanup

Avoid ongoing charges. Restore the CPU target to 50–60% if retaining the lab. Otherwise delete the ASG, launch template, manual instances, ALB and target group. Also check for remaining EBS volumes, Elastic IP addresses, snapshots and NAT gateways.

To pause the ASG before deleting it, set desired, minimum and maximum capacity to zero. This terminates ASG-managed instances.

Result

The test demonstrated scale out from two to as many as four instances, scale in after load ended, automatic replacement following a health check failure and automatic registration of ASG instances with the ALB target group.
