Lab 44 - Application Load Balancer in the Custom VPC

Objective

Deploy an Internet-facing Application Load Balancer inside the existing Custom VPC, associate it with a Target Group, register an EC2 instance as a target, and validate end-to-end connectivity.

This lab begins the migration from the Default VPC to the Custom VPC architecture that will eventually host the Auto Scaling Group.

The Application Load Balancer becomes the only Internet-facing component. The EC2 instance is now protected by allowing HTTP traffic only from the ALB Security Group.

New CloudFormation Resources

Resource Purpose

LabApplicationLoadBalancer Internet-facing Application Load Balancer deployed across both public subnets.
LabAlbTargetGroup Registers the EC2 instance and performs health checks.
LabAlbListener Listens on HTTP port 80 and forwards requests to the Target Group.

Security Improvements

Before this lab, the EC2 Security Group allowed inbound HTTP traffic from anywhere: 

0.0.0.0/0

After the update, inbound HTTP traffic is allowed only from the ALB Security Group: 

SourceSecurityGroupId: LabAlbSecurityGroup

This follows AWS security best practices by preventing direct Internet access to application servers.

Enterprise Architecture

Enterprise environments rarely expose EC2 instances directly to the Internet. Instead, Internet traffic always enters through a Load Balancer.

Advantages include:

  • High Availability
  • Load Distribution
  • Automatic Health Checks
  • Integration with Auto Scaling
  • Centralized TLS termination
  • Improved Security
  • Simplified application scaling

Lessons Learned

  • An Internet-facing ALB must be deployed inside public subnets.
  • Application servers should never be directly exposed to the Internet.
  • Security Groups can reference other Security Groups.
  • Target Groups continuously monitor application health.
  • CloudFormation Change Sets reduce deployment risk.
  • The ALB provides the foundation for scalable architectures.
