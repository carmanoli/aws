Lab 17 - Amazon SNS + AWS Lambda

1. Lab Objective

The objective of this lab is to integrate an existing Amazon SNS topic with an existing AWS Lambda function. This lab demonstrates how SNS can be used as an event distribution service and how Lambda can process messages published to an SNS topic.

6. Step 2 - Update the Lambda Function Code

Inside the Lambda function, navigate to:

Code > Code source

Replace the existing code with the following Python code:

import json

def lambda_handler(event, context):
    print("Full event received:")
    print(json.dumps(event, indent=2))

    for record in event.get("Records", []):
        sns_message = record.get("Sns", {}).get("Message", "No message found")
        sns_subject = record.get("Sns", {}).get("Subject", "No subject")

        print("SNS subject:", sns_subject)
        print("SNS message:", sns_message)

    return {
        "statusCode": 200,
        "body": json.dumps("SNS message processed successfully")
    }

Click:

Deploy

Technical Explanation

When Amazon SNS invokes a Lambda function, the message is delivered inside the event object. The event contains a Records array because AWS event sources often support batch-style delivery. Each SNS notification appears as one record inside this array.

The relevant path for the SNS message is:

event["Records"][0]["Sns"]["Message"]

The relevant path for the SNS subject is:

event["Records"][0]["Sns"]["Subject"]

Technical Explanation

10. Understanding the SNS Event Structure

The SNS event delivered to Lambda contains several important fields.

Field Description
Records Array containing one or more event records.
EventSource Identifies the source service. For SNS, the value is aws:sns.
Sns Object containing the SNS notification details.
TopicArn ARN of the SNS topic that published the message.
Subject Optional subject of the SNS message.
Message Main message payload sent through SNS.
MessageAttributes Optional metadata that can be used for filtering or routing.

11. Why This Architecture Matters

This lab introduces a basic event-driven architecture pattern. Instead of one service directly calling another service, an event is published to SNS and then delivered to one or more subscribers. This creates a more flexible and decoupled design.

For example, the same SNS topic can later send messages to:

  • Email subscribers
  • Lambda functions
  • Amazon SQS queues
  • HTTPS endpoints
  • Other event-processing systems

12. Professional Best Practices

Reuse Existing Topics When Appropriate

The lab reused the existing lab-cloudwatch-topic instead of creating a new SNS topic. This is useful when the same notification should be consumed by multiple systems.

Use Least Privilege Permissions

SNS should only be allowed to invoke the Lambda function that needs to process the message. AWS automatically adds a resource-based policy to the Lambda function when the trigger is configured through the console.

Log the Full Event During Learning and Troubleshooting

Printing the full event is useful in a lab environment because it helps understand the structure of the payload. In production, logs should avoid exposing sensitive data.

Design for Multiple Records

The Lambda function loops through event.get("Records", []) instead of assuming only one message. This is a safer and more reusable pattern for event-driven workloads.

Prepare for Future Integrations

This setup prepares the environment for future labs involving Amazon EventBridge, Amazon S3 event notifications, SQS queues, and more advanced automation patterns.
