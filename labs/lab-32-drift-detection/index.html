Lab 32 - AWS CloudFormation Drift Detection

This lab introduces AWS CloudFormation Drift Detection, a feature used to verify whether deployed infrastructure still matches the CloudFormation template.

Lab Objectives

  • Understand Configuration Drift.
  • Execute Drift Detection.
  • Interpret Drift Detection results.
  • Understand the meaning of IN_SYNC and MODIFIED.
  • Restore infrastructure consistency.
  • Apply Infrastructure as Code best practices.

Prerequisites

  • CloudFormation stack lab26-asg-stack.
  • Auto Scaling Group managed by CloudFormation.
  • One running EC2 instance.
  • Launch Template version 2.
  • Application Load Balancer.
  • Target Group.

Component Name
CloudFormation Stack lab26-asg-stack

What is Configuration Drift?

CloudFormation assumes that the template is the single source of truth. Whenever a resource is modified manually using the AWS Console, AWS CLI, or API, its actual configuration may no longer match the template. This situation is called Configuration Drift.

Drift Detection Workflow

  1. Deploy infrastructure using CloudFormation.
  2. Modify a managed resource manually.
  3. Run Drift Detection.
  4. Review the detected differences.
  5. Restore the resource to the expected state.
  6. Verify the stack returns to IN_SYNC.

Step 1 - Verify Stack Status

The CloudFormation stack was confirmed to be operational before testing.

Status: 

UPDATE_COMPLETE

Step 2 - Simulate Configuration Drift

A manual tag was added directly to the Auto Scaling Group.

Key Value
DriftTest Manual

This modification was intentionally performed outside CloudFormation.

Step 3 - Execute Drift Detection

CloudFormation → Stack Actions → Detect Drift

CloudFormation compared the deployed infrastructure against the template.

Step 4 - Analyze Drift Results

The stack status changed to: 

DRIFTED

The affected resource was: 

AWS::AutoScaling::AutoScalingGroup
Logical ID: LabWebAutoScalingGroup

CloudFormation detected differences in the resource tags.

Observed Differences

Property Expected Current
Tags Name = lab-cfn-web-instance DriftTest = Manual

After removing the manual tag, CloudFormation still reported drift because the original Name tag had also been removed.

This demonstrates that editing tags manually may unintentionally replace the complete user-defined tag set.

Step 5 - Restore Expected Configuration

The missing CloudFormation tag was recreated.

Key Value Propagate At Launch
Name lab-cfn-web-instance True

Step 6 - Execute Drift Detection Again

After restoring the expected configuration, Drift Detection was executed again.

Final status: 

IN_SYNC

CloudFormation Drift States

Status Description
IN_SYNC The deployed resource matches the CloudFormation template.
MODIFIED The resource exists but differs from the template.
DELETED The resource was manually deleted outside CloudFormation.
NOT_CHECKED The resource type does not support Drift Detection or was not evaluated.

Lessons Learned

  • CloudFormation templates are the source of truth.
  • Manual changes introduce Configuration Drift.
  • Drift Detection identifies property-level differences.
  • Drift Detection does not automatically repair infrastructure.
  • Infrastructure should be corrected by restoring the expected configuration or by updating the CloudFormation template.
  • Avoid manual modifications to CloudFormation-managed resources whenever possible.

Best Practices

  • Manage infrastructure exclusively through Infrastructure as Code.
  • Run Drift Detection periodically.
  • Investigate every detected drift before applying updates.
  • Keep CloudFormation templates under version control.
  • Restrict manual console changes in production environments.
