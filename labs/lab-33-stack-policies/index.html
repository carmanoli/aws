Lab 33 - AWS CloudFormation Stack Policies

Objective

Learn how CloudFormation Stack Policies protect critical infrastructure resources against accidental updates while still allowing controlled maintenance using Temporary Stack Policies.

Prerequisites

  • AWS CloudFormation Stack already managing the Auto Scaling Group.
  • CloudShell available.
  • Existing stack:
    • lab26-asg-stack

Architecture Before the Lab

  • CloudFormation Stack
  • Launch Template
  • Auto Scaling Group
  • Application Load Balancer
  • Target Group
  • Systems Manager Session Manager
  • CloudTrail
  • CloudWatch

Step 1 - Create the Stack Policy

Create the following JSON file.

{
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "LogicalResourceId/LabWebAutoScalingGroup"
    },
    {
      "Effect": "Allow",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "*"
    }
  ]
}

Step 2 - Apply the Stack Policy

The current AWS Console no longer exposes Stack Policy management for existing stacks. Therefore the policy was applied using AWS CloudShell. 

aws cloudformation set-stack-policy \
  --stack-name lab26-asg-stack \
  --stack-policy-body file://lab33-stack-policy.json

The command completed successfully and the permanent Stack Policy became active.

Step 3 - Test the Protection

A CloudFormation Update Stack was executed changing:

Parameter Old Value New Value
DesiredCapacity 1 2

Expected result: 

Action denied by stack policy:
Statement does not allow Update:* for
LogicalResourceId/LabWebAutoScalingGroup

CloudFormation automatically rolled back the operation.

Final stack status: 

UPDATE_ROLLBACK_COMPLETE

Step 4 - Create a Temporary Stack Policy

Create the following JSON file.

{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "Update:*",
      "Principal": "*",
      "Resource": "LogicalResourceId/LabWebAutoScalingGroup"
    }
  ]
}

Step 5 - Perform a Controlled Update

aws cloudformation update-stack \
  --stack-name lab26-asg-stack \
  --use-previous-template \
  --parameters \
    ParameterKey=DesiredCapacity,ParameterValue=2 \
    ParameterKey=MinSize,UsePreviousValue=true \
    ParameterKey=MaxSize,UsePreviousValue=true \
  --stack-policy-during-update-body file://lab33-temporary-stack-policy.json

The update completed successfully.

The Auto Scaling Group increased its desired capacity to two instances.

Step 6 - Restore the Environment

The desired capacity was returned to one instance using the same Temporary Stack Policy. 

aws cloudformation update-stack \
  --stack-name lab26-asg-stack \
  --use-previous-template \
  --parameters \
    ParameterKey=DesiredCapacity,ParameterValue=1 \
    ParameterKey=MinSize,UsePreviousValue=true \
    ParameterKey=MaxSize,UsePreviousValue=true \
  --stack-policy-during-update-body file://lab33-temporary-stack-policy.json

Stack Policies vs IAM Policies

IAM Policy Stack Policy
Controls what users may do. Controls what CloudFormation may modify.
Applies account-wide. Applies only to one CloudFormation Stack.
Protects AWS services. Protects CloudFormation-managed resources.
Always evaluated. Evaluated only during Stack Updates.

Enterprise Use Cases

  • Protect production databases.
  • Prevent accidental replacement of Auto Scaling Groups.
  • Protect Load Balancers during application deployments.
  • Require explicit authorization for infrastructure modifications.
  • Allow temporary exceptions during controlled maintenance windows.

Key Learning Outcomes

  • Understand the purpose of Stack Policies.
  • Create permanent Stack Policies.
  • Protect critical resources.
  • Observe blocked CloudFormation updates.
  • Use Temporary Stack Policies safely.
  • Restore infrastructure after maintenance.
  • Understand how Stack Policies complement IAM.

Final Infrastructure Status

  • CloudFormation Stack operational.
  • Permanent Stack Policy enabled.
  • Auto Scaling Group protected.
  • Desired Capacity restored to 1.
  • Single EC2 instance running.
  • No additional AWS costs introduced.

Conclusion

This lab demonstrated one of the most important operational safety mechanisms available in AWS CloudFormation. Stack Policies protect business-critical resources against accidental modifications while Temporary Stack Policies allow controlled maintenance without permanently weakening infrastructure protection. These techniques are widely used in production enterprise environments.
